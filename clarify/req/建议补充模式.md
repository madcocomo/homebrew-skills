# 目标
此文档是我根据经验认为可能有帮助的模式，简要在下面说明。作为生成结果的参考和补充。但研究本身不应该局限于此。同时本文档中的内容也较为简略和粗糙，需要研究过程进行补充和完善。

# 建议补充模式
- 流程图。适用于描述流程和步骤的内容。注意适用范围。可以和信息分层结合使用。
- 泳道图。类似于流程图，但适用于描述多角色协作的流程和步骤。
- 序列图。适用于描述系统交互和消息传递的内容。
- 状态机。适用于描述较为复杂的系统状态变化和事件响应的内容。
- 判定矩阵。适用于描述多条件决策和选择的内容。容易发现遗漏的条件和选项。
- 实例化。适用于描述抽象概念和公式等内容。通过具体实例消除理解模糊发现歧义。
- 极端示例。实例化的特例，使用边界值或者极大极小的值产生例子，通过极端取值让潜在谬误更明显。
- 反向验证。通过以反命题的形式表达原本内容的反面，验证原本信息的正确性。也有利于发现正向表达时遗漏的边界情况。
- 场景化描述。适用于多处零散但关联的信息或改动。通过构建一个具体场景，把零散信息放入场景中进行描述，帮助理解和发现遗漏。
- 可执行验证。当信息内容本身可以被执行时，使用工具或者代码执行验证。通过外部的验证发现信息中的错误和遗漏。
