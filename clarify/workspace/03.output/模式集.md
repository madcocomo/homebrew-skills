# AI信息呈现模式集

## 导言

### 背景

随着AI辅助软件开发越来越普及，人类需要理解和验证AI生成的内容（如需求文档、设计方案、代码、测试用例等）。然而，AI的输出往往存在以下问题：
- 信息组织方式不利于人类理解
- 缺乏便于人类发现错误的结构
- 抽象描述导致理解歧义

### 核心原则

**本模式集聚焦于"信息展示方式"，而非"AI生成方式"。**

| 正确方向 ✅ | 错误方向 ❌ |
|-----------|-----------|
| 信息展示方式 | 提示词改进 |
| 帮助人类理解 | 帮助AI更可靠 |
| 帮助人类发现错误 | 减少AI错误 |
| 具体可操作 | 泛泛而谈 |

### 模式分类

本模式集包含三大类共10个模式：

1. **可视化表示模式**（4个）：通过图形化方式呈现信息
2. **检验发现模式**（4个）：帮助人类主动发现错误
3. **综合应用模式**（2个）：综合性的信息组织方式

---

## 一、可视化表示模式

### 1.1 流程图模式

**分类**: 可视化表示

#### 目的

将流程性信息以图形化方式呈现，使人类能够一目了然地理解步骤顺序、分支逻辑和整体流程结构。

#### 问题根源

- 人类工作记忆容量有限，无法一次性记住多个步骤的顺序
- 自然语言描述的流程容易产生顺序误解
- 复杂流程中的分支条件难以用文字清晰表达
- 人类对视觉化的流程理解效率远高于文字描述

#### 描述

流程图模式要求将过程性、步骤性的信息以图形化方式呈现，包括：
- 明确的步骤节点
- 步骤之间的箭头指向
- 条件分支的判断节点
- 循环结构的标识

#### 适用场景

- AI生成的需求处理流程
- AI生成的数据处理流水线
- AI生成的操作步骤说明
- 任何包含"先做什么、后做什么"的内容

#### 实施方法

1. 识别信息中的步骤序列
2. 确定每个步骤的输入和输出
3. 识别条件分支和并行分支
4. 使用标准流程图符号：
   - 矩形：普通步骤
   - 菱形：判断/条件
   - 箭头：流程方向
   - 圆角矩形：开始/结束
5. 确保每个节点有明确的含义

#### 示例

**正面示例**：

```
用户注册流程：
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   开始      │────▶│  填写信息   │────▶│  验证信息   │
└─────────────┘     └─────────────┘     └─────────────┘
                                                  │
                            ┌─────────────────────┤
                            ▼                     ▼
                     ┌─────────────┐       ┌─────────────┐
                     │  验证通过   │       │  验证失败   │
                     └─────────────┘       └─────────────┘
                            │                     │
                            ▼                     ▼
                     ┌─────────────┐       ┌─────────────┐
                     │  创建账户   │       │  返回修正   │
                     └─────────────┘       └─────────────┘
                            │
                            ▼
                     ┌─────────────┐
                     │   结束      │
                     └─────────────┘
```

**反面示例**：

"用户注册流程：首先用户填写信息，然后系统验证信息，如果验证通过就创建账户，如果验证失败就返回让用户修正信息，最后结束。"

（问题：人类需要自己脑补流程结构，容易产生理解偏差）

#### 效果

- **理解效率**：人类可在几秒内理解原本需要阅读大量文字才能理解的流程
- **错误发现**：流程图使步骤遗漏、顺序错误、分支不完整等问题一目了然

#### 组合建议

- 可与**判定矩阵模式**组合使用：流程中的判断节点可以用判定矩阵进一步说明
- 可与**实例化模式**组合使用：每个步骤的具体操作可以通过实例化进一步说明

### 1.2 泳道图模式

**分类**: 可视化表示

#### 目的

将多角色协作的流程以泳道图方式呈现，使人类能够清晰看到每个角色的职责边界和交互接口。

#### 问题根源

- 人类难以同时跟踪多个角色的行为和责任
- 跨角色的信息传递容易产生理解盲区
- 角色间的接口和依赖关系难以用线性文字描述

#### 描述

泳道图是流程图的一种变体，通过纵向的"泳道"区分不同角色，每个角色在自己的泳道内执行操作，跨角色的箭头表示信息传递。

#### 适用场景

- AI生成的涉及多角色交互的业务流程
- AI生成的系统架构设计
- AI生成的协作流程说明
- 任何包含"谁做什么"的内容

#### 实施方法

1. 识别流程中的参与角色
2. 确定每个角色的职责范围
3. 将每个角色的操作放入对应的泳道
4. 用箭头表示跨角色的信息传递
5. 标注每个交互点的输入输出

#### 示例

**正面示例**：

```
┌─────────────┬─────────────┬─────────────┐
│   用户      │    客服     │    系统     │
├─────────────┼─────────────┼─────────────┤
│             │             │             │
│  提交请求   │             │             │
│     ───────▶│             │             │
│             │  接收请求   │             │
│             │     ──────▶│             │
│             │             │  处理请求   │
│             │             │     ◀────── │
│             │  返回结果   │             │
│     ◀──────│             │             │
│             │             │             │
└─────────────┴─────────────┴─────────────┘
```

**反面示例**：

"用户提交请求给客服，客服接收后转交给系统处理，系统处理完成后返回结果给客服，客服再将结果反馈给用户。"

（问题：人类需要自己脑补角色边界和交互接口）

#### 效果

- **理解效率**：清晰展示角色边界和交互关系
- **错误发现**：角色职责不清、接口遗漏等问题一目了然

#### 组合建议

- 可与**序列图模式**组合使用：泳道图展示整体流程，序列图展示详细交互
- 可与**判定矩阵模式**组合使用：各角色的决策条件可以用判定矩阵说明

### 1.3 序列图模式

****分类**: 可视化表示

#### 目的

将系统交互和消息传递以时序图方式呈现，使人类能够清晰理解组件间的调用顺序和数据流动。

#### 问题根源

- 人类难以在脑中维护多个组件的交互时序
- 异步调用和回调的时序关系容易产生混淆
- 数据在系统间流动的路径需要显式表达才能清晰

#### 描述

序列图（时序图）是一种强调时间顺序的交互图，横轴表示不同的组件对象，纵轴表示时间，箭头表示消息传递，垂直方向表示时间流逝。

#### 适用场景

- AI生成的API调用说明
- AI生成的组件交互设计
- AI生成的消息队列流程
- 任何包含"谁调用谁、传什么数据"的内容

#### 实施方法

1. 识别参与交互的组件/服务
2. 确定消息传递的时序
3. 标注每条消息的内容（参数、返回值）
4. 标识异步消息和同步消息
5. 标注关键的时间点或状态变化

#### 示例

**正面示例**：

```
用户              前端              后端              数据库
 │                 │                 │                 │
 │   1. 提交请求   │                 │                 │
 │────────────────▶│                 │                 │
 │                 │   2. 验证请求    │                 │
 │                 │────────────────▶│                 │
 │                 │                 │   3. 查询数据   │
 │                 │                 │────────────────▶│
 │                 │                 │◀────────────────│
 │                 │◀────────────────│                 │
 │                 │                 │   4. 返回结果   │
 │◀────────────────│                 │                 │
 │                 │                 │                 │
```

**反面示例**：

"用户向前端提交请求，前端验证后调用后端API，后端查询数据库并返回结果给前端。"

（问题：消息传递的具体内容和时序不够清晰）

#### 效果

- **理解效率**：组件间的调用关系和时间顺序一目了然
- **错误发现**：调用顺序错误、遗漏调用、参数错误等问题容易发现

#### 组合建议

- 可与**泳道图模式**组合使用：序列图展示详细时序，泳道图展示角色职责
- 可与**实例化模式**组合使用：具体的请求/响应内容可以通过实例化说明

### 1.4 状态机模式

**分类**: 可视化表示

#### 目的

将系统状态变化和事件响应以状态机方式呈现，使人类能够清晰理解状态的合法转移路径。

#### 问题根源

- 人类难以枚举一个系统可能处于的所有状态
- 状态间的转移条件容易被遗漏
- 非法状态转移可能导致严重的逻辑错误

#### 描述

状态机模式用图形化方式展示系统的所有可能状态、状态间的转移条件以及触发转移的事件。

#### 适用场景

- AI生成的业务状态流转规则
- AI生成的工作流设计
- AI生成的生命周期管理说明
- 任何包含"什么条件下从A状态到B状态"的内容

#### 实施方法

1. 枚举系统的所有可能状态
2. 确定每个状态的初始/终止属性
3. 识别状态间的转移事件和条件
4. 标注每个转移的触发条件和动作
5. 标识非法/不可能的状态转移

#### 示例

**正面示例**：

```
                    ┌──────────────────┐
                    ▼                  │
              ┌───────────┐      ┌──────┴─────┐
              │  待支付   │─────▶│  已支付    │
              └───────────┘      └────────────┘
                    ▲                  │
                    │      ┌───────────┴───────────┐
                    │      │                       │
                    │      ▼                       ▼
              ┌───────────┐              ┌────────────┐
              │  已取消   │◀─────────────│  已发货    │
              └───────────┘              └────────────┘
                    │                       │
                    └───────────────────────┘
                         (交易完成)

转移条件：
- 待支付 → 已支付：用户完成支付
- 已支付 → 已发货：商家发货
- 已支付 → 已取消：用户取消/超时
- 已发货 → 交易完成：用户确认收货
- 已取消 → 交易完成：不可能转移
- 已发货 → 已取消：不可能转移
```

**反面示例**：

"订单有待支付、已支付、已发货、已完成等状态，用户支付后变成已支付，商家发货后变成已发货，用户确认收货后变成已完成。"

（问题：状态转移路径不完整，缺少取消路径，非法转移未标注）

#### 效果

- **理解效率**：完整的状态空间和转移路径一目了然
- **错误发现**：遗漏的状态、缺失的转移条件、非法转移等问题容易发现

#### 组合建议

- 可与**判定矩阵模式**组合使用：状态转移条件可以用判定矩阵详细说明
- 可与**极端示例模式**组合使用：可以用极端状态验证状态机的完整性

## 二、检验发现模式

### 2.1 判定矩阵模式

**分类**: 检验发现

#### 目的

将多维度条件与结果的对应关系以矩阵形式呈现，使人类能够系统性地检查所有条件组合，容易发现遗漏。

#### 问题根源

- 人类在处理多条件组合时容易遗漏某些组合
- 条件数量超过工作记忆容量时，判断准确性急剧下降
- 自然语言描述的条件组合容易产生歧义

#### 描述

判定矩阵将多个条件的可能取值作为维度，列出所有可能的条件组合，并标注每种组合对应的结果或行为。

#### 适用场景

- AI生成的多条件业务规则
- AI生成的权限控制规则
- AI生成的定价规则
- 任何包含"如果...那么..."的规则描述

#### 实施方法

1. 识别规则涉及的所有条件
2. 确定每个条件的可能取值
3. 生成所有条件组合的矩阵
4. 为每种组合标注对应的结果
5. 标识不可能或无效的组合
6. 检查是否有遗漏的结果

#### 示例

**正面示例**：

**用户权限判定矩阵**：

| 用户类型 | 订单状态 | 是否有权限 | 备注 |
|---------|---------|-----------|------|
| 普通用户 | 待支付 | ✅ 有 | 可查看自己的订单 |
| 普通用户 | 已支付 | ✅ 有 | 可查看自己的订单 |
| 普通用户 | 已发货 | ✅ 有 | 可查看自己的订单 |
| 普通用户 | 已完成 | ✅ 有 | 可查看自己的订单 |
| VIP用户 | 待支付 | ✅ 有 | 可查看所有订单 |
| VIP用户 | 已支付 | ✅ 有 | 可查看所有订单 |
| VIP用户 | 已发货 | ✅ 有 | 可查看所有订单 |
| VIP用户 | 已完成 | ✅ 有 | 可查看所有订单 |
| 管理员 | 待支付 | ✅ 有 | 可查看和修改所有订单 |
| 管理员 | 已支付 | ✅ 有 | 可查看和修改所有订单 |
| 管理员 | 已发货 | ⚠️ 有限制 | 可查看，但修改需审批 |
| 管理员 | 已完成 | ❌ 无 | 不可修改已完成订单 |

（效果：任何人都可以快速检查是否有遗漏的组合）

**反面示例**：

"普通用户可以查看自己的订单，VIP用户可以查看所有订单，管理员可以修改未完成的订单。"

（问题：规则描述不完整，遗漏了管理员对已发货订单的处理规则）

#### 效果

- **理解效率**：所有条件组合一目了然，无需自己组合
- **错误发现**：容易发现条件组合遗漏、结果冲突、规则矛盾

#### 组合建议

- 可与**流程图模式**组合使用：矩阵用于定义规则，流程图用于展示处理逻辑
- 可与**实例化模式**组合使用：每个组合的具体效果可以用实例化说明

### 2.2 实例化模式

**分类**: 检验发现

#### 目的

通过具体的实例来说明抽象概念，消除理解歧义，帮助人类发现抽象描述中的隐藏问题。

#### 问题根源

- 抽象描述存在多种可能的理解方式
- 人类倾向于选择最符合自己预期的理解方式
- 抽象与具体之间的映射往往不是一对一的

#### 描述

实例化模式要求对任何抽象概念、规则、定义都提供至少一个具体的实例，让实例与抽象描述相互印证。

#### 适用场景

- AI生成的概念定义
- AI生成的规则说明
- AI生成的数据结构说明
- 任何包含抽象术语的内容

#### 实施方法

1. 识别描述中的抽象概念和术语
2. 为每个抽象概念提供一个或多个具体实例
3. 确保实例能够覆盖概念的主要含义
4. 标注实例与概念的对应关系
5. 检查实例是否可能导致误解

#### 示例

**正面示例**：

"用户等级规则：
- VIP用户：累计消费满10000元的用户

实例说明：
- 用户A，累计消费9500元 → ❌ 不是VIP
- 用户B，累计消费10000元 → ✅ 是VIP
- 用户C，累计消费50000元 → ✅ 是VIP

边界情况：
- 刚好10000元时升级为VIP"

**反面示例**：

"用户等级规则：累计消费满10000元的用户为VIP用户。"

（问题：人类可能对"累计消费"的计算方式、是否包含退款等细节产生不同理解）

#### 效果

- **理解效率**：具体实例帮助快速理解抽象概念
- **错误发现**：实例与抽象描述不一致时，人类容易发现歧义

#### 组合建议

- 可与**判定矩阵模式**组合使用：矩阵的每个单元格都是一个实例
- 可与**极端示例模式**组合使用：边界情况的实例是极端示例

### 2.3 极端示例模式

**分类**: 检验发现

#### 目的

通过边界值、极限情况等极端示例，让潜在的错误、遗漏或不合理之处更容易暴露。

#### 问题根源

- 正常情况下的规则往往看起来合理
- 极端情况更容易暴露规则的漏洞
- 人类倾向于验证正常情况而忽略边界情况

#### 描述

极端示例模式要求对每个规则、逻辑、计算都测试边界值和极限情况，确保规则在极端情况下仍然合理。

#### 适用场景

- AI生成的计算公式
- AI生成的验证规则
- AI生成的数量限制
- 任何可能受边界影响的逻辑

#### 实施方法

1. 识别规则中的数值边界
2. 构造边界值和极端值测试用例
3. 验证规则在极端情况下是否仍然合理
4. 检查极端情况下是否有异常行为
5. 识别规则中的潜在漏洞

#### 示例

**正面示例**：

"订单折扣规则：满100元减10元，满200元减25元"

极端示例测试：
- 订单金额：99.99元 → 实际支付：99.99元（未达门槛）
- 订单金额：100元 → 实际支付：90元（刚好达标）
- 订单金额：100.01元 → 实际支付：90.01元（超出1分钱，多减1分钱？）
- 订单金额：200元 → 实际支付：175元（刚好达标）
- 订单金额：200.01元 → 实际支付：175.01元（是否有更好的折扣？）

**发现的问题**：
- 100-199.99元区间的订单没有最优解
- 建议优化为"每满100元减10元"

**反面示例**：

"订单折扣规则：满100元减10元，满200元减25元"

（问题：没有测试极端情况，人类可能不会发现100-199区间的效率问题）

#### 效果

- **理解效率**：极端情况帮助理解规则的完整行为
- **错误发现**：边界漏洞、计算错误、逻辑缺陷容易在极端示例中暴露

#### 组合建议

- 可与**实例化模式**组合使用：极端示例是一种特殊的实例
- 可与**反向验证模式**组合使用：极端情况可以用于反向验证

---

### 2.4 反向验证模式

**分类**: 检验发现

#### 目的

通过验证命题的反面是否成立，帮助发现正向描述中的遗漏、歧义或错误。

#### 问题根源

- 人类的验证倾向：倾向于验证"对"的情况，忽略"错"的情况
- 正向描述往往只关注"应该是什么"，忽略"不应该是什么"
- 遗漏边界条件和反面情况是常见的错误

#### 描述

反向验证模式要求对每个正向命题都思考其反面，检查：
- 反面是否也成立
- 反面情况是否被遗漏
- 正向和反面是否存在矛盾

#### 适用场景

- AI生成的规则描述
- AI生成的约束条件
- AI生成的验证逻辑
- 任何包含"如果...那么..."的内容

#### 实施方法

1. 提取描述中的正向命题
2. 将命题改写为否定形式
3. 检查否定命题是否成立
4. 识别被遗漏的反面情况
5. 检查正反两面是否存在矛盾

#### 示例

**正面示例**：

**正向描述**："只有VIP用户才能享受免费配送"

反向验证：
- 非VIP用户不能享受免费配送 ✅
- VIP用户可以享受免费配送 ✅

**但如果描述是**："VIP用户可以享受免费配送"

反向验证：
- 非VIP用户呢？是否可以有条件享受？
- VIP用户是否可能有例外情况？
- 免费配送是否有金额门槛？

**发现问题**：原描述不完整，遗漏了非VIP用户的情况和例外条件

**反面示例**：

"VIP用户可以享受免费配送。"

（问题：人类可能不会主动思考"非VIP用户怎么办"这类问题）

#### 效果

- **理解效率**：通过正反对比，更全面理解规则
- **错误发现**：遗漏的边界条件、例外情况容易暴露

#### 组合建议

- 可与**判定矩阵模式**组合使用：矩阵可以同时包含正向和反向情况
- 可与**极端示例模式**组合使用：极端情况可以用于反向验证

---

## 三、综合应用模式

### 3.1 场景化描述模式

**分类**: 综合应用

#### 目的

通过构建具体的场景故事，将零散的知识点串联起来，帮助人类在真实上下文中理解信息。

#### 问题根源

- 零散的知识点难以形成整体认知
- 人类难以将抽象规则与实际应用联系起来
- 缺乏上下文的描述容易产生理解偏差

#### 描述

场景化描述模式要求为信息构建一个具体的场景，将抽象的规则、概念放入场景中说明。

#### 适用场景

- AI生成的功能说明
- AI生成的业务流程
- AI生成的业务规则
- 任何需要将多个知识点串联起来的内容

#### 实施方法

1. 识别需要说明的核心信息
2. 构建一个合理的场景背景
3. 将每个知识点融入场景叙事
4. 确保场景能够覆盖主要情况
5. 标注场景与实际应用的对应关系

#### 示例

**正面示例**：

"订单处理系统规则说明：

场景：张三周一下单购买手机，周三收到货

1. 订单创建（周一）
   - 张三在APP上下单，订单号：OD20240101001
   - 订单金额：5000元
   - 订单状态：待支付

2. 支付完成（周一）
   - 张三通过微信支付完成付款
   - 订单状态：已支付

3. 发货（周二）
   - 商家通过顺丰发货，快递单号：SF1234567890
   - 订单状态：已发货

4. 收货（周三）
   - 张三确认收货
   - 订单状态：已完成

通过这个场景，您可以理解：
- 订单状态的完整生命周期
- 每个状态的触发条件
- 状态变更的时效"

**反面示例**：

"订单处理系统规则：
- 待支付 → 已支付：用户完成支付
- 已支付 → 已发货：商家发货
- 已发货 → 已完成：用户确认收货
- 已支付 → 已取消：用户取消/超时"

（问题：缺乏具体场景，人类难以将规则与实际流程对应）

#### 效果

- **理解效率**：场景帮助将抽象规则具体化
- **错误发现**：场景中的逻辑漏洞、遗漏步骤容易暴露

#### 组合建议

- 可与**流程图模式**组合使用：场景描述配合流程图
- 可与**实例化模式**组合使用：场景中的具体数据是实例

---

### 3.2 可执行验证模式

**分类**: 综合应用

#### 目的

当信息内容本身可以被执行时，通过实际运行来验证信息的正确性，发现并改正错误和遗漏。

#### 问题根源

- 静态的文字描述难以验证正确性
- 复杂的逻辑关系容易产生理解错误
- 某些错误只有在实际执行时才能暴露
- 人类自行执行验证成本高，容易遗漏

#### 描述

可执行验证模式要求，当信息可以被执行时：
1. **优先AI直接执行**：如果AI工具当前有可使用的工具或命令（如代码执行、命令行运行等），AI应该直接执行验证
2. **呈现验证结果**：将执行结果呈现给人类，包括预期结果和实际结果的对比
3. **自行改正问题**：如果发现错误，在人类确认后自行改正
4. **仅在无法直接验证时**：才提供可执行的形式和说明，让人类自行验证

**核心原则**：尽可能降低人类的验证成本，让AI承担验证的执行工作。

#### 适用场景

- AI生成的代码示例（可以直接运行验证）
- AI生成的配置文件（可以解析验证格式）
- AI生成的计算公式（可以执行验证结果）
- AI生成的脚本（可以实际运行）
- 任何可以被执行的描述

#### 实施方法

**步骤一：判断是否可以执行验证**
1. 识别信息中可以被执行的部分
2. 判断当前AI工具是否有可使用的执行工具（代码执行、命令行、API调用等）
3. 如果可以执行，执行步骤二；否则执行步骤三

**步骤二：AI直接执行验证（优先）**
1. 将需要验证的内容提取为可执行的格式
2. 准备测试用例（包括正常情况、边界情况）
3. 直接执行验证
4. 对比预期结果与实际结果
5. 发现问题时，自行修正并重新验证
6. 向人类呈现：
   - 验证执行命令/代码
   - 执行结果
   - 发现的问题（如有）
   - 已执行的修正（如有）

**步骤三：提供执行验证（无法直接验证时）**
1. 将执行部分提取为可运行的格式
2. 提供测试用例
3. 说明预期的执行结果
4. 标注可能导致错误的地方
5. 告知用户如何执行验证

#### 示例

**正面示例（AI直接执行验证）**：

"订单折扣计算规则：满100元减10元，满200元减25元。

我将其实现为代码并执行验证：

```python
def calculate_discount(order_amount):
    if order_amount >= 200:
        return order_amount * 0.875  # 200减25，等于8.75折
    elif order_amount >= 100:
        return order_amount * 0.9     # 100减10，等于9折
    else:
        return order_amount

# 测试验证
test_cases = [
    (50, 50),
    (100, 90),
    (150, 135),  # 原规则是140，这里有问题
    (200, 175),
]

for amount, expected in test_cases:
    actual = calculate_discount(amount)
    status = "✅" if actual == expected else "❌"
    print(f"{status} 金额: {amount}, 预期: {expected}, 实际: {actual}")
```

执行结果：
```
✅ 金额: 50, 预期: 50, 实际: 50
✅ 金额: 100, 预期: 90, 实际: 90
❌ 金额: 150, 预期: 135, 实际: 135 （注：150元按9折是135元，但用户可能误以为应该享受200元档的优惠）
❌ 金额: 200, 预期: 175, 实际: 175
```

**发现的问题**：
- 150元订单实际上没有更优解（按9折是135元，按200元档8.75折是131.25元，但150元不满足200元门槛）
- 建议优化规则为"每满100元减10元"，这样150元可以减15元，实付135元，与当前一致，但200元可以减20元更划算

**我已修正规则为**：每满100元减10元（不足100元不减）

```python
def calculate_discount(order_amount):
    discount = (order_amount // 100) * 10
    return order_amount - discount

# 重新验证
test_cases = [
    (50, 50),     # 不满100，不减
    (100, 90),    # 100减10
    (150, 135),   # 100减10，等于9折
    (200, 180),   # 200减20，比之前175更划算
    (250, 220),   # 200减20，等于8.8折
]
```

现在规则更加合理。"

**反面示例（仅告知用户，未执行验证）**：

"订单折扣计算规则：满100元减10元，满200元减25元。

您可以运行以下代码验证规则是否正确：

```python
def calculate_discount(order_amount):
    if order_amount >= 200:
        return order_amount * 0.875
    elif order_amount >= 100:
        return order_amount * 0.9
    else:
        return order_amount
```

（问题：AI只是提供了可执行的形式，但没有实际执行验证，也没有发现规则中的潜在问题）

#### 效果

- **理解效率**：AI直接执行验证，人类无需自己运行代码即可理解规则的实际行为
- **错误发现**：运行时的实际输出暴露逻辑错误，且AI可以自行发现并指出问题
- **自动修正**：AI可以自行修正发现的错误并重新验证，降低人类处理成本

#### 组合建议

- 可与**极端示例模式**组合使用：用极端值作为测试用例进行验证
- 可与**实例化模式**组合使用：测试用例本身就是实例
- 可与**反向验证模式**组合使用：在执行验证时同时进行反向验证

## 四、应用指南

### 4.1 模式选择流程

```
                    ┌─────────────────────┐
                    │  需要呈现什么类型   │
                    │     的信息？         │
                    └──────────┬──────────┘
                               │
          ┌────────────────────┼────────────────────┐
          ▼                    ▼                    ▼
   ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
   │   流程性信息  │    │   规则性信息  │    │   概念性信息  │
   │ (步骤、分支)  │    │ (条件、结果)  │    │ (定义、关系)  │
   └──────┬───────┘    └──────┬───────┘    └──────┬───────┘
          │                    │                    │
          ▼                    ▼                    ▼
   ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
   │   流程图     │    │   判定矩阵   │    │   实例化     │
   │   泳道图     │    │   极端示例   │    │   场景化描述  │
   │   序列图     │    │   反向验证   │    │              │
   │   状态机     │    │   可执行验证 │    │              │
   └──────────────┘    └──────────────┘    └──────────────┘
```

### 4.2 模式组合建议

| 信息类型 | 推荐模式组合 | 说明 |
|---------|------------|------|
| 业务流程 | 流程图 + 场景化描述 | 流程图展示结构，场景化帮助理解 |
| 多角色协作 | 泳道图 + 序列图 + 判定矩阵 | 泳道图展示角色，序列图展示时序，判定矩阵说明规则 |
| 状态管理 | 状态机 + 极端示例 + 判定矩阵 | 状态机展示状态，判定矩阵说明转移条件，极端示例测试 |
| 业务规则 | 判定矩阵 + 实例化 + 反向验证 | 判定矩阵展示规则，实例化说明含义，反向验证发现遗漏 |
| 算法逻辑 | 流程图 + 可执行验证 + 极端示例 | 流程图展示逻辑，可执行验证确认正确性，极端示例测试边界 |

### 4.3 常见场景推荐

| 场景 | 推荐模式 |
|-----|---------|
| 需求文档中的业务流程 | 流程图 + 场景化描述 |
| 设计文档中的系统交互 | 序列图 + 泳道图 |
| 代码审查 | 可执行验证 + 极端示例 |
| 业务规则说明 | 判定矩阵 + 实例化 + 反向验证 |
| 状态变更说明 | 状态机 + 判定矩阵 |
| 权限规则说明 | 判定矩阵 + 极端示例 |
| 接口说明 | 序列图 + 实例化 |
| 数据结构说明 | 实例化 + 场景化描述 |
