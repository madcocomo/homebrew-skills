## 新增模式清单

### PAT-015: 自文档化模式

**分类**: 可理解性增强 - 信息结构层

**目的**: 在代码生成过程中自动添加代码注释和文档，使代码自解释，降低理解和维护成本。

**描述**:
自文档化模式借鉴了软件工程中的"代码即文档"理念和认知科学的组块化原理。该模式要求人工智能在生成代码时，不仅输出可执行的代码，还要自动添加必要的注释、文档字符串和类型标注，使代码结构一目了然。

这一模式的核心要素包括：
- 意图注释：说明代码模块的目的和用途
- 参数注释：说明输入输出参数的意义和约束
- 决策注释：说明关键逻辑决策的原因
- 示例注释：提供使用示例代码

**适用场景**:
- 复杂业务逻辑的代码生成
- API接口代码的生成
- 需要他人维护或协作的代码
- 学习用途的代码示例生成

**实施指南**:
1. 在生成代码前，先说明代码的整体结构和模块划分
2. 为每个函数/类添加文档字符串（docstring），说明功能、参数、返回值
3. 为关键逻辑添加行内注释，说明"为什么"而非"是什么"
4. 添加类型注解（type hints），明确输入输出类型
5. 在文件头部添加使用说明和依赖信息

**示例**:

*正面示例*:
```python
"""
用户认证模块

提供基于JWT的用户身份验证功能，支持登录、令牌刷新和权限验证。

主要功能：
- 用户登录验证
- JWT令牌生成和验证
- 令牌刷新机制
- 权限检查装饰器
"""

class AuthService:
    """认证服务类"""

    def __init__(self, secret_key: str, token_expiry: int = 3600):
        """
        初始化认证服务

        Args:
            secret_key: 用于JWT签名密钥，需妥善保管
            token_expiry: 令牌有效期（秒），默认1小时
        """
        self.secret_key = secret_key
        self.token_expiry = token_expiry

    def authenticate(self, username: str, password: str) -> Optional[Dict]:
        """
        验证用户凭据

        Args:
            username: 用户名（不区分大小写）
            password: 明文密码

        Returns:
            认证成功返回用户信息和JWT令牌，失败返回None
        """
        # 查找用户（先验证用户存在性，避免Timing攻击）
        user = self._find_user(username)
        if not user:
            return None

        # 验证密码（使用常量时间比较防止时序攻击）
        if not self._verify_password(password, user['password_hash']):
            return None

        # 生成令牌
        token = self._generate_token(user['id'], user['roles'])
        return {'user': user, 'token': token}
```

**注意事项**:
- 注释应该说明"为什么"，而不是重复代码在做什么
- 避免过度注释，清晰的代码本身应该是自解释的
- 文档字符串应保持更新，避免误导
- 遵循目标语言的文档约定（如Python的PEP 257）

**与其他模式的关系**:
- 依赖: 结构化输出模式 - 自文档化是结构化代码的具体实现
- 增强: 格式标准化模式 - 文档格式应符合项目规范
- 冲突: 简洁性要求 - 过多注释可能影响代码简洁性

### PAT-016: 执行验证模式

**分类**: 错误发现 - 交互反馈层

**目的**: 在代码生成后提供执行验证和测试建议，帮助用户发现代码中的潜在错误。

**描述**:
执行验证模式要求人工智能在生成代码后，提供代码的执行验证信息和测试建议。这一模式借鉴了软件测试中的"测试驱动开发"理念和形式化方法中的模型检测技术。

该模式的核心要素包括：
- 语法验证：确认代码语法正确
- 依赖检查：确认所需依赖可用
- 单元测试建议：提供基本测试用例
- 边界条件提示：提醒可能的边界情况

**适用场景**:
- 生产环境代码的生成
- 复杂算法的代码生成
- 需要可靠运行的 critical 代码
- 新手开发者的代码辅助

**实施指南**:
1. 生成代码后，进行基本的语法检查并报告结果
2. 列出代码所需的所有依赖及其版本要求
3. 提供至少2-3个基本测试用例，覆盖正常路径
4. 识别代码中的边界条件（如空输入、极大值、特殊字符等）
5. 提供代码运行的基本指导（如何执行、参数配置等）

### PAT-017: 依赖关系说明模式

**分类**: 可理解性增强 - 信息结构层

**目的**: 明确说明代码中的依赖关系，帮助用户理解代码的上下文依赖和调用关系。

**描述**:
依赖关系说明模式要求人工智能在生成代码时，说明代码的依赖关系和上下文依赖。这一模式源于软件工程中的"依赖注入"思想和认知科学中的"组块化"原理。

该模式的核心要素包括：
- 外部依赖：代码所依赖的库、模块、API
- 内部依赖：代码内部的模块调用关系
- 数据依赖：输入数据的格式和约束
- 环境依赖：运行所需的环境配置

**适用场景**:
- 多模块项目的代码生成
- 需要集成外部服务的代码
- 有复杂数据流处理的代码
- 需要环境配置的代码

**实施指南**:
1. 在代码开头明确列出所有外部依赖（库、版本要求）
2. 在关键调用点说明被调用函数的职责
3. 说明输入数据的预期格式和约束条件
4. 提供环境配置要求（环境变量、配置文件等）
5. 如有替代方案，说明可替换的依赖

### PAT-018: 代码复杂度控制模式

**分类**: 可理解性增强 - 表达策略层

**目的**: 控制代码生成的复杂度，使其符合用户的理解能力水平，避免生成过于复杂的代码。

**描述**:
代码复杂度控制模式借鉴了认知科学中的工作记忆容量理论（7±2组块），要求人工智能根据用户需求和场景，控制生成代码的复杂度。

该模式的核心要素包括：
- 复杂度分级：根据用户需求提供不同复杂度的代码
- 渐进增强：从简单实现开始，逐步添加高级特性
- 模块拆分：将复杂逻辑拆分为多个可理解的模块
- 替代方案：提供简单和复杂两种实现供选择

**适用场景**:
- 新手开发者的代码教学
- 快速原型开发
- 需要维护的正式项目
- 不同技术水平的用户

**实施指南**:
1. 首先确认用户的技术水平或期望的代码复杂度级别
2. 提供至少两个复杂度级别的代码：
   - 简洁版：核心功能，易于理解
   - 完整版：包含所有特性，复杂度较高
3. 复杂代码要明确标注可简化的部分
4. 提供复杂度说明，帮助用户理解权衡
5. 为复杂代码提供注释版的简化说明

**示例**:

*正面示例*:
```python
# 根据复杂度级别生成不同代码

# ===== 简洁版（推荐新手使用）=====
def process_data(data):
    """简单的数据处理：过滤空值并转为大写"""
    return [str(x).upper() for x in data if x]

# ===== 完整版（包含完整错误处理）=====
def process_data(data: List[Any], *, strict: bool = False) -> List[str]:
    """
    完整的数据处理函数

    Args:
        data: 输入数据列表
        strict: 严格模式，为True时抛出异常而非返回空列表

    Returns:
        处理后的字符串列表

    Raises:
        TypeError: 当strict=True且输入类型不支持时
    """
    if not isinstance(data, (list, tuple)):
        if strict:
            raise TypeError(f"Expected list/tuple, got {type(data).__name__}")
        return []

    result = []
    for item in data:
        try:
            if item is None or (isinstance(item, str) and not item.strip()):
                continue
            result.append(str(item).upper())
        except (ValueError, TypeError):
            if strict:
                raise
            continue

    return result
```

- 复杂度说明要清晰，便于用户选择
- 简单版本应该是正确的，不应该包含bug

### PAT-019: 代码审查提示模式

**分类**: 错误发现 - 表达策略层

**目的**: 在生成代码后提供代码审查视角的提示，帮助用户发现潜在的代码问题和改进点。

**描述**:
代码审查提示模式借鉴了软件工程中代码审查（Code Review）的最佳实践，要求人工智能从代码审查者的角度审视生成的代码，提供改进建议。

该模式的核心要素包括：
- 安全提示：可能的安全漏洞
- 性能提示：可能的性能问题
- 可维护性提示：代码结构问题
- 最佳实践提示：不符合编码规范的地方

**适用场景**:
- 生产环境代码的生成
- 需要遵循编码规范的团队项目
- 安全敏感的应用
- 性能关键的应用

**实施指南**:
1. 从安全性角度审查代码，标注可能的漏洞
2. 检查性能相关的问题（如循环中的重复计算）
3. 指出可维护性问题（如过长的函数、重复代码）
4. 检查是否遵循目标语言的编码规范
5. 提供具体的改进建议和替代代码

**示例**:

*正面示例*:
```python
# 代码审查提示

## 原始代码
def get_user(user_id):
    user = db.query(f"SELECT * FROM users WHERE id = {user_id}")
    return user

## 审查发现

### ⚠️ 安全问题：SQL注入漏洞
原始代码直接拼接用户输入到SQL查询中，这是严重的SQL注入漏洞。

建议修改为：
```python
def get_user(user_id):
    user = db.query(
        "SELECT * FROM users WHERE id = %s",
        (user_id,)  # 参数化查询
    )
    return user
```

### ⚠️ 性能问题：可能返回大量数据
使用SELECT * 可能获取不必要的字段。

建议修改为：
```python
def get_user(user_id):
    user = db.query(
        "SELECT id, username, email, created_at FROM users WHERE id = %s",
        (user_id,)
    )
    return user
```

### 💡 可维护性建议
- 考虑添加返回类型注解
- 建议添加日志记录，便于调试
```

*反面示例*:
```

```

**注意事项**:
- 审查建议要有针对性，不要泛泛而谈
- 严重问题要明确标注级别（如Security, Performance）
- 建议应该是可操作的

**与其他模式的关系**:
- 增强: 边界标注模式 - 审查提示也是代码能力边界的一部分
