# 目标、场景和问题定义

## 核心目标

产生一套模式集合，用例澄清和展现信息、内容等，用来帮助人们更好理解和发现错误。这套模式预期使用在AI辅助的软件开发过程中，比如SDD类的工作流程，但不限于某种特定方法论或者工作风格。

## 背景理解

随着AI辅助开发能力的不断提升，软件开发的成本分布和效率瓶颈已经发生了翻转。在传统的开发模式中，构建出具备一定质量标准的软件系统已经隐含地证明了开发者或相关干系人对系统的理解达到了相适应的水平。然而，AI辅助构建能力的提升，意味着产生达到质量标准的代码不再是整个链条的瓶颈，相反对问题的澄清、对结果的验证、对实现机制的理解越来越制约整体效率。亦即所谓"理解债务"。

这个概念受启发自传统开发中的"技术债务"但又有显著的区别。技术债务指自觉或不自觉地牺牲实现代码/架构的质量换取构建工作量减少，从而得到最终的更快交付、更快验证等结果。而理解债务在现阶段更多是指不自觉地产生对需求、实现、结果等方面的理解不足，得到"更快"的错觉。

基于这样的理解，提升人理解系统的效率相比于提升AI构建系统的能力越来越重要，成为提升整体效率的关键杠杆。这套模式的目标就是在整个过程的各个环节为人类理解提供支持。

## 典型场景

- **需求澄清**: AI生成的需求文档需要人类理解其含义、范围和限制，确保符合预期
- **设计评审**: AI生成的设计方案需要人类评审其合理性
- **代码审查**: AI生成的代码需要人类检查其正确性、可维护性和安全性
- **测试用例生成**: AI生成的测试用例需要人类理解其覆盖范围和有效性
- **代码/架构理解**: 人类需要理解现有代码（无论它是由人类或AI产生的）的架构和实现细节，以便进行维护和扩展
- **结果展示**: AI生成的报告、数据分析结果等需要人类理解其含义和应用场景

## 三方面的基本限制

### 人的限制

1. 人难以，事实上也不可能，一次性完整表达系统相关的所有信息
2. 人类的认知能力有限，面对复杂信息时容易产生认知过载，难以有效处理和理解大量信息
3. 人类在不同状态下，不同的人在面临相同问题时可能有显著差异的表述，理解程度也有较大差异
4. 人类不容易理解哪些隐含的知识需要明确表达，即"知识的诅咒"
5. 由于自然语言的模糊性，人类在理解信息时会自以为理解了，但事实上并未真正进行校验

### AI的限制

1. 不同AI模型，模型的不同版本在相同问题的表现上会有显著差异。这些差异是和具体模型实现高度相关的。人无法按照一般化的原理说明来弥合实现的差异达到一套提示词/指令在所有相当算力的模型上获得一致结果
2. AI的输出，擅长保持连贯性（coherent）；小范围的一致性（consistent）也不错但较弱；对于结果内部逻辑正确性差距较大，虽有一定提高但本质上不具备稳定可信的逻辑推理校验， 对于外部世界相关的正确性完全无法保证
3. AI不擅长处理整体与局部的"匀称性"。往往由于短期具体聚焦的任务目标，产生喧宾夺主的结果
4. 当整体上下文超出AI上下文窗口时，分批次产生的结果难以保证一致性，容易产生重复

### 软件开发复杂性的限制

1. 产生有价值的可用软件的过程，不是一个一次性的线性过程，而往往需要反复多次看到结果后意识到下列各种差异：设想目标与实际可行目标的差异，设想目标与对目标的表述的差异，目标表述与解决方案匹配度的差异，解决方案的表述与实际方案的差异。并在不断修正差异中得到收敛的结果
2. AI辅助开发的效率来源于AI的先验知识。即AI可以理解很多不言自明的常识。而辅助开发中表现不良往往是AI先验能力与当前上下文的不匹配。这种不匹配是高度上下文相关的，无法通过一般化的补充原则、知识等弥合

## 关键定义

### 信息呈现（Information Presentation）

本系统所关注的"信息呈现"特指：**AI在输出内容时，如何组织、结构化、标注和表达信息的方式**。

这与以下概念有本质区别：

- **提示词优化（Prompt Optimization）**: 用户（人）如何改进给AI的指令，以获得更好的输出
- **AI自我改进（AI Self-Improvement）**: AI如何减少错误、提高质量
- **用户验证技巧（User Verification Skills）**: 人如何检查和验证AI的输出

### 核心原则

1. **模式指导的是AI的行为**，而非人的行为
2. **模式关注的是输出形式**，而非输入指令
3. **模式的目标是让输出更容易被理解**，而非让AI更少出错
