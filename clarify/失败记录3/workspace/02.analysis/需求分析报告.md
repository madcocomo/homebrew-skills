# 需求分析报告

## 一、核心定义

### 1.1 信息呈现定义

本系统所关注的"信息呈现"特指：**AI在输出内容时，如何组织、结构化、标注和表达信息的方式**。

这与以下概念有本质区别：
- **提示词优化（Prompt Optimization）**: 用户（人）如何改进给AI的指令，以获得更好的输出
- **AI自我改进（AI Self-Improvement）**: AI如何减少错误、提高质量
- **用户验证技巧（User Verification Skills）**: 人如何检查和验证AI的输出

**核心原则**：
1. **模式指导的是AI的行为**，而非人的行为
2. **模式关注的是输出形式**，而非输入指令
3. **模式的目标是让输出更容易被理解**，而非让AI更少出错

### 1.2 关键概念

#### 理解债务

理解债务是受启发自传统开发中的"技术债务"但又有显著区别的概念：
- **技术债务**：指自觉或不自觉地牺牲实现代码/架构的质量换取构建工作量减少，从而得到最终的更快交付、更快验证等结果
- **理解债务**：在现阶段更多是指不自觉地产生对需求、实现、结果等方面的理解不足，得到"更快"的错觉

随着AI辅助构建能力的提升，产生达到质量标准的代码不再是整个链条的瓶颈，相反对问题的澄清、对结果的验证、对实现机制的理解越来越制约整体效率。理解债务问题因此变得更加突出。

#### 人的限制

人类在理解和沟通方面存在以下基本限制：
1. 人难以，事实上也不可能，一次性完整表达系统相关的所有信息
2. 人类的认知能力有限，面对复杂信息时容易产生认知过载，难以有效处理和理解大量信息
3. 人类在不同状态下，不同的人在面临相同问题时可能有显著差异的表述，理解程度也有较大差异
4. 人类不容易理解哪些隐含的知识需要明确表达，即"知识的诅咒"
5. 由于自然语言的模糊性，人类在理解信息时会自以为理解了，但事实上并未真正进行校验

#### AI的限制

AI在输出方面存在以下基本限制：
1. 不同AI模型，模型的不同版本在相同问题的表现上会有显著差异。这些差异是和具体模型实现高度相关的。人无法按照一般化的原理说明来弥合实现的差异达到一套提示词/指令在所有相当算力的模型上获得一致结果
2. AI的输出，擅长保持连贯性（coherent）；小范围的一致性（consistent）也不错但较弱；对于结果内部逻辑正确性差距较大，虽有一定提高但本质上不具备稳定可信的逻辑推理校验，对于外部世界相关的正确性完全无法保证
3. AI不擅长处理整体与局部的"匀称性"。往往由于短期具体聚焦的任务目标，产生喧宾夺主的结果
4. 当整体上下文超出AI上下文窗口时，分批次产生的结果难以保证一致性，容易产生重复

## 二、区分标准

### 2.1 信息呈现 vs 提示词优化

| 特征 | "信息呈现"模式 | "提示词优化"内容 |
|------|---------------|-----------------|
| 主语 | AI（系统）应该... | 用户应该... |
| 焦点 | 输出如何组织 | 输入如何改进 |
| 目的 | 使输出更容易被理解 | 获得更好的输出 |
| 时机 | AI生成时应用 | 用户提问时应用 |
| 指导对象 | AI行为 | 用户行为 |

**典型错误示例**：
> 实施指南:
> 1. 在系统提示中加入推理引导，如"请一步步思考"或"展示你的推理过程"
> 2. 对于复杂问题，要求分步骤解决

这是指导用户如何写提示词，而不是指导AI如何呈现信息。

**正确方向**：直接告诉AI："当你需要展示推理过程时，使用以下格式..."

### 2.2 信息呈现 vs AI自我改进

**核心区别**：
- 信息呈现关注的是"如何更好地呈现信息以便人发现错误"
- AI自我改进关注的是"如何让AI更少产生错误"

**典型错误示例**：
> 注意事项:
> - 推理过程要真实反映思考过程，不要为了形式而编造步骤
> - 推理过程应与最终结论保持一致，避免过程和结果矛盾
> - 推理过程本身也可能出错，需要结合一致性验证

这是告诉AI"如何减少错误"，而不是"如何呈现信息以便人发现错误"。

**正确方向**："当你在呈现推理过程时，应该..."

### 2.3 正确与错误的模式示例对比

**正确的"信息呈现"模式示例**：
```markdown
## 模式：推理过程呈现

### 目的
当AI需要展示推理过程时，使用统一的结构化格式，使人类能够：
- 快速理解AI是如何得出结论的
- 有效地检验推理链条的合理性
- 在发现错误时定位具体步骤

### 呈现格式
当需要展示推理时，采用以下结构：

## 推理过程

### 前提确认
[列出所有使用的前提条件]

### 步骤 1: [步骤名称]
**理由**: [为什么进行这个步骤]
**结果**: [这个步骤得出什么]

### 步骤 2: [步骤名称]
**理由**: [为什么进行这个步骤]
**结果**: [这个步骤得出什么]

### 结论
[基于以上步骤，最终结论是什么]
```

**错误的"提示词优化"示例**：
```markdown
## 提示词技巧：如何让AI展示推理

在提问时加上：
- "请一步步思考"
- "展示你的推理过程"
- "解释你是如何得出这个结论的"
```

## 三、研究问题

### 3.1 核心研究问题

1. **什么才是真正的"信息呈现"模式？**
   - 如何确保模式指导的是AI的行为（而非用户的行为）
   - 如何确保模式关注的是输出形式（而非输入指令）
   - 如何确保模式的目标是让输出更容易被理解（而非让AI更少出错）

2. **之前的模式为什么偏离了目标？**
   - 模式变成了"提示词指南"：指导用户如何写提示词
   - 模式变成了"AI自我改进指南"：告诉AI如何减少错误
   - 混淆了"呈现"与"验证"：把AI如何提供溯源和人如何验证来源混在一起

3. **如何确保重构后的模式聚焦于呈现方式？**
   - 建立明确的区分标准
   - 在设计模式时始终问："这是AI的行为还是用户的行为？"
   - 在设计模式时始终问："这是输出形式还是输入指令？"

### 3.2 次要研究问题

1. **信息呈现与用户验证技巧的边界如何界定？**
   - 信息呈现是AI的行为
   - 用户验证技巧是人的行为
   - 需要明确区分两者，避免混为一谈

2. **如何处理不同AI模型之间的差异？**
   - 不同模型的表现有显著差异
   - 模式需要具有一定程度的通用性，同时承认模型特定优化的必要性

3. **在典型场景中如何应用信息呈现模式？**
   - 需求澄清：AI生成的需求文档需要人类理解其含义、范围和限制
   - 设计评审：AI生成的设计方案需要人类评审其合理性
   - 代码审查：AI生成的代码需要人类检查其正确性、可维护性和安全性
   - 测试用例生成：AI生成的测试用例需要人类理解其覆盖范围和有效性
   - 代码/架构理解：人类需要理解现有代码的架构和实现细节
   - 结果展示：AI生成的报告、数据分析结果等需要人类理解其含义和应用场景
