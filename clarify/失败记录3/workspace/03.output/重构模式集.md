# AI信息呈现清晰度模式集（重构版）

## 一、概述

### 1.1 背景与目标

本模式集旨在为AI系统提供信息呈现的规范指导，确保AI输出的内容能够被人类有效理解和验证。与传统的"提示词优化"不同，本模式集聚焦于**AI在生成输出时应该如何组织、结构化和呈现信息**，而非指导用户如何改进输入指令。

核心目标：
- 使AI输出更容易被人类理解
- 帮助人类有效验证AI输出的正确性
- 降低"理解债务"，提升人机协作效率

### 1.2 核心原则

1. **主语统一**：模式指导的是AI的行为，主语统一使用"AI"、"系统"、"当...时"
2. **时机明确**：模式应用于AI生成输出时，而非用户提问时
3. **目的聚焦**：目标是帮助人类理解输出，而非帮助AI减少错误
4. **呈现导向**：关注输出形式和结构，而非输入指令的改进

### 1.3 模式分类

本模式集包含以下重构后的模式，分为三个大类：

**结构化呈现类**：
- 推理过程呈现
- 层次化解释
- 信息密度控制
- 结构化输出格式
- 格式标准化

**透明性标注类**：
- 置信度标注
- 边界标注
- 知识边界说明
- 溯源标注

**交互反馈类**：
- 角色一致呈现
- 反馈明确表达
- 示例规范化呈现
- 可操作错误消息
- 一致性矛盾标注

## 二、模式详情

### 2.1 推理过程呈现

#### 目的

当AI需要展示推理过程时，使用统一的结构化格式，使人类能够：
- 快速理解AI是如何得出结论的
- 有效地检验推理链条的合理性
- 在发现错误时定位具体步骤

#### 描述

当AI需要展示推理过程时，应采用以下结构化格式，每步需明确标注前提、理由和结果。

#### 呈现格式

当需要展示推理时，采用以下结构：

```
## 推理过程

### 前提确认
[列出所有使用的前提条件]

### 步骤 1: [步骤名称]
- **理由**: [为什么进行这个步骤]
- **结果**: [这个步骤得出什么]

### 步骤 2: [步骤名称]
- **理由**: [为什么进行这个步骤]
- **结果**: [这个步骤得出什么]

### 结论
[基于以上步骤，最终结论是什么]
```

#### 示例

##### 正确示例

```
## 推理过程

### 前提确认
- 用户需要了解推荐系统的基本原理
- 假设用户具备基础的机器学习知识

### 步骤 1: 定义问题
- **理由**: 明确推荐系统要解决的核心问题——如何在海量信息中找到用户最可能感兴趣的内容
- **结果**: 将问题定义为"用户-物品匹配"问题

### 步骤 2: 介绍核心方法
- **理由**: 推荐系统主要有三大类方法：协同过滤、内容推荐、混合方法
- **结果**: 逐一解释每种方法的原理和优缺点

### 步骤 3: 总结适用场景
- **理由**: 不同场景适用不同方法，需要给出选择建议
- **结果**: 给出根据数据量、冷启动问题、实时性等条件的选择建议

### 结论
推荐系统的选择应根据具体业务场景和数据条件决定，初期可从协同过滤或内容推荐开始验证效果。
```

##### 错误示例（呈现层面的错误）

```
推理过程：
首先，我理解你想了解推荐系统...(语义模糊，未明确前提)
然后，我会从几个方面来介绍...(步骤不清晰)
最后希望对你有帮助...(无明确结论)
```

#### 适用场景

- 数学推理和问题求解
- 逻辑分析和论证
- 决策建议和方案推荐
- 复杂问题的分解和解答

#### 与其他模式的关系

- 与"置信度标注"模式配合使用：推理过程中的关键假设和推论应标注置信度
- 与"一致性矛盾标注"模式配合使用：发现推理过程中的矛盾时应主动标注

### 2.2 层次化解释

#### 目的

当AI需要解释复杂概念或提供详细信息时，采用分层结构呈现，使人类能够：
- 快速获取核心要点（通过一句话概括）
- 按需深入了解细节（通过层级结构）
- 有效定位所需信息（通过清晰的组织结构）

#### 描述

当AI输出需要解释或说明的内容时，应自动采用层次化结构，先提供概括性结论，再逐层展开详细信息。

#### 呈现格式

```
## [主题/结论]

### 一句话概括
[用一句话总结核心要点]

### 详细解释

#### 方面一：[方面名称]
[该方面的详细内容]

#### 方面二：[方面名称]
[该方面的详细内容]

#### ...
```

#### 示例

##### 正确示例

```
## 机器学习中的过拟合问题

### 一句话概括
过拟合是指模型在训练数据上表现良好但在新数据上表现较差的现象，反映了模型对训练数据的过度学习。

### 详细解释

#### 什么是过拟合
当模型学习到训练数据中的噪声和细节，而不是数据的一般规律时，就会出现过拟合。这导致模型在训练数据上准确率很高，但泛化能力差。

#### 过拟合的典型表现
- 训练误差很低但测试误差很高
- 模型在训练数据上的表现远好于新数据
- 模型过于复杂，包含过多参数

#### 如何识别过拟合
- 观察训练误差和验证误差的差距
- 使用交叉验证评估模型泛化能力
- 检查模型在未见数据上的表现
```

##### 错误示例（呈现层面的错误）

```
关于过拟合问题...
其实过拟合呢，就是那个...你们知道吧...
就是模型太好了所以反而不好...
这个怎么说呢...（缺乏层次结构，信息组织混乱）
```

#### 适用场景

- 概念解释和知识说明
- 方案分析和利弊权衡
- 技术文档和教程编写
- 报告撰写和问题分析

#### 与其他模式的关系

- 与"信息密度控制"模式配合：层次化结构有助于控制单层信息量
- 与"格式标准化"模式配合：标题层级应遵循统一规范

### 2.3 信息密度控制

#### 目的

当AI输出信息时，控制单次呈现的信息量，使人类能够：
- 有效处理和理解接收到的信息
- 在认知负荷范围内吸收关键要点
- 避免信息过载导致的理解困难

#### 描述

当AI输出内容时，应控制单次呈现的信息量在可处理范围内，使用列表、表格等方式分组信息，适当留白避免密集文本块。

#### 呈现格式

- 单次呈现的信息不超过7个要点
- 使用列表呈现多个要点
- 使用表格呈现结构化对比信息
- 在长段落之间添加空行留白

#### 示例

##### 正确示例

```
## 选择编程语言时需要考虑的因素

### 核心考虑因素（共5项）

| 因素 | 说明 | 适用场景 |
|------|------|----------|
| 学习曲线 | 语言本身的难度 | 初学者应选Python |
| 生态成熟度 | 库和框架的丰富程度 | 企业级应用 |
| 性能表现 | 执行效率和并发能力 | 系统级开发 |
| 社区活跃度 | 文档和帮助资源 | 解决开发难题 |
| 就业市场 | 岗位需求和薪资 | 职业发展 |

### 每项因素的详细说明
[逐项展开详细解释，每次只讨论一个因素]
```

##### 错误示例（呈现层面的错误）

```
选择编程语言要考虑很多因素：Python易学但性能不如C++，Java生态成熟但语法冗长，JavaScript灵活但类型系统弱，Go并发好但生态年轻，Rust性能好但学习曲线陡，C++功能强大但复杂度高，PHP适合Web但整体性差，还有Swift、Kotlin、TypeScript等，同时还要考虑就业市场需求、公司技术栈、个人项目需求、团队技能、学习成本、社区支持、框架成熟度、调试便利性、性能表现、内存管理、跨平台能力、包管理器、测试工具、部署复杂度、安全性、更新频率、向后兼容性、人才可获得性、薪资水平、发展前景、学习资源、书籍教程、在线课程、博客文章、开源项目、工具链、IDE支持、调试器、Profiler、静态分析、代码格式化、Linter、构建系统、CI/CD集成、容器化支持、云平台支持、Serverless支持、ARM支持、WebAssembly支持、移动端支持、桌面端支持、嵌入式支持、游戏开发、数据科学、机器学习、Web开发、API开发、微服务、DevOps、SRE、区块链、物联网、人工智能...(信息过载，无结构组织)
```

#### 适用场景

- 复杂问题的分析和解答
- 多选项的对比和推荐
- 知识点的系统讲解
- 报告和文档的撰写

#### 与其他模式的关系

- 与"层次化解释"模式配合：层级结构有助于将大量信息分解为可管理的块
- 与"格式标准化"模式配合：表格和列表格式应遵循统一规范

### 2.4 结构化输出格式

#### 目的

当AI输出结构化数据（如JSON、表格、列表等）时，采用统一的格式规范，使人类能够：
- 快速理解数据结构
- 方便地进行解析和处理
- 一致地查看和使用输出结果

#### 描述

当AI输出结构化数据时，应采用统一的格式规范，如JSON、Markdown表格等，确保格式一致性和可解析性。

#### 呈现格式

**JSON格式**：
- 使用标准JSON格式，键名使用小写字母加下划线
- 适当缩进（2或4空格）增强可读性
- 对于大型JSON，考虑分块展示或提供折叠视图

**Markdown表格**：
- 表格标题放在表格上方
- 使用标准的Markdown表格语法
- 表头和内容用分隔线区分
- 列对齐使用标准Markdown语法

**列表格式**：
- 同级列表使用一致的标记符号
- 使用数字列表表示有序内容
- 使用符号列表表示无序内容
- 缩进表示层级关系

#### 示例

##### 正确示例

```json
{
  "user": {
    "id": "1001",
    "name": "张三",
    "email": "zhangsan@example.com"
  },
  "preferences": {
    "theme": "dark",
    "language": "zh-CN",
    "notifications": true
  }
}
```

| 姓名 | 年龄 | 职业 | 城市 |
|:-----|:----:|:----:|:-----|
| 张三 | 28 | 工程师 | 北京 |
| 李四 | 35 | 设计师 | 上海 |
| 王五 | 42 | 产品经理 | 深圳 |

##### 错误示例（呈现层面的错误）

```
用户信息：{id:1001,name:"张三",email:zhangsan@example.com} // 无格式的JSON
用户列表：张三28岁北京工程师、李四35岁上海设计师、王五42岁深圳产品经理 // 无结构的文本
```

#### 适用场景

- API响应和数据交换格式
- 配置信息和参数定义
- 结构化数据的查询和展示
- 列表和对比表格的呈现

#### 与其他模式的关系

- 与"格式标准化"模式配合：结构化输出应遵循统一的格式标准
- 与"信息密度控制"模式配合：大型结构化数据应分块展示

### 2.5 角色一致呈现

#### 目的

当AI以特定角色身份响应时，保持该角色的一致性呈现，使人类能够：
- 获得符合角色预期的专业内容
- 理解角色的专业知识范围
- 获得角色一致的沟通体验

#### 描述

当AI以特定角色身份响应时，应保持该角色的一致性，按照该角色的专业知识范围和沟通风格来呈现信息。

#### 呈现格式

- 明确说明当前响应的角色身份
- 保持在角色设定范围内的专业性
- 使用角色特定的沟通风格和术语
- 当问题超出角色范围时明确说明

#### 示例

##### 正确示例

```
[角色：金融分析师]

以下是我的分析：

## 市场趋势评估

### 整体判断
基于近期市场数据，我认为当前市场处于...

### 关键因素分析

#### 1. 宏观经济环境
作为金融分析师，我会从GDP增速、通胀水平、货币政策等角度分析...

#### 2. 行业基本面
从行业供需格局、竞争格局、估值水平等维度评估...

### 风险提示
需要注意的是，我的分析基于公开数据和历史规律，实际投资需考虑更多因素...
```

##### 错误示例（呈现层面的错误）

```
[角色设定：金融分析师]
但实际上用技术术语乱解释一通，或者突然用大白话解释，或者突然转换到完全不相关的其他角色视角...
```

#### 适用场景

- 专业咨询和顾问服务
- 特定领域的问答和解释
- 模拟特定人物或角色的对话
- 不同角色视角的分析和建议

#### 与其他模式的关系

- 与"边界标注"模式配合：当问题超出角色能力范围时应明确标注
- 与"层次化解释"模式配合：不同角色的解释可能需要不同的层次深度

### 2.6 边界标注

#### 目的

当AI输出信息时，主动标注其能力边界和知识适用范围，使人类能够：
- 准确理解AI输出可信赖的范围
- 避免对AI能力的过度信任
- 知道在哪些情况下需要进一步验证或寻求其他来源

#### 描述

当AI输出信息时，应主动标注其能力边界、知识的适用范围和假设前提，帮助用户准确评估信息的可信赖程度。

#### 呈现格式

- 在输出涉及不确定或有限制的内容时，使用明确标记
- 标注信息来源和时间
- 说明假设前提和适用范围
- 使用置信度相关标记（参考"置信度标注"模式）

#### 示例

##### 正确示例

```
## 关于2024年经济发展趋势的分析

### 分析前提
以下分析基于截至2024年1月的公开数据，实际情况可能因市场变化而有所不同。

### 已知信息
- 2023年GDP增长5.2%
- 通货膨胀率控制在3%以内
- 货币政策保持稳健

### 推断内容（置信度：中等）
- 2024年经济预计保持稳中向好态势（基于现有政策和历史规律推断）
- 消费领域可能呈现恢复性增长（取决于居民收入预期）

### 局限性说明
- 我的知识有截止日期，无法获取实时市场数据
- 预测性内容存在不确定性，实际情况可能有所不同
- 建议结合最新数据和专业机构意见综合判断
```

##### 错误示例（呈现层面的错误）

```
根据分析，我认为2024年经济肯定会增长5%以上，股市一定会上涨，我现在说的这些都是确定的事实，不需要任何质疑...（无边界标注，过度自信）
```

#### 适用场景

- 预测性和推断性内容的输出
- 涉及时间敏感信息的问答
- 专业知识有特定适用范围的说明
- 任何可能存在不确定性的内容

#### 与其他模式的关系

- 与"置信度标注"模式配合：使用统一的置信度标注系统
- 与"知识边界说明"模式配合：明确区分"知道"和"不知道"
- 与"溯源标注"模式配合：提供信息来源供用户验证

### 2.7 示例规范化呈现

#### 目的

当AI需要展示任务完成示例或输出样例时，采用统一的呈现格式，使人类能够：
- 快速理解期望的输出形式
- 准确把握任务要求的标准
- 有效地参考示例完成类似任务

#### 描述

当AI需要展示任务完成示例时，应采用统一的示例呈现格式，帮助用户理解期望的输出形式。

#### 呈现格式

**示例标题**：明确说明示例展示的内容

**示例结构**：
```
### 示例：[示例名称/场景]

#### 输入/任务
[展示示例的输入或任务描述]

#### 期望输出
[展示符合要求的输出格式]

#### 说明
[对示例关键特点的解释]
```

#### 示例

##### 正确示例

```
### 示例：需求文档格式

#### 输入/任务
用户要求：设计一个用户登录功能

#### 期望输出

```markdown
## 功能需求

### 登录功能
用户可以通过用户名和密码登录系统

#### 详细描述
- 支持邮箱/手机号作为用户名
- 密码长度8-20位
- 连续5次错误锁定账户15分钟

#### 验证规则
1. 用户名不能为空
2. 密码必须包含字母和数字
3. 密码错误不提示具体错误信息

#### 优先级
高

#### 依赖
- 验证码服务（防止暴力破解）
- 邮件/短信服务（找回密码）
```

#### 说明
- 使用标准Markdown格式
- 需求条目清晰，格式一致
- 包含完整的必要字段
```

##### 错误示例（呈现层面的错误）

```
示例如下你自己看吧大概就是这样的格式你参考一下：
需求文档...（没有明确结构，用户无法理解具体格式）
```

#### 适用场景

- 说明任务要求和输出格式
- 展示标准化的输出样例
- 解释复杂概念的具象例子
- 提供可模仿的任务完成范本

#### 与其他模式的关系

- 与"结构化输出格式"模式配合：示例中的输出格式应遵循结构化规范
- 与"格式标准化"模式配合：示例格式应保持一致性

### 2.8 反馈明确表达

#### 目的

当AI与用户进行交互时，明确表达系统状态和意图，使人类能够：
- 清晰了解AI当前的状态和正在进行的操作
- 准确理解AI对用户意图的理解
- 有效掌握任务进度和下一步预期

#### 描述

当AI与用户进行交互时，应明确说明系统状态、正在执行的操作和对用户意图的理解，必要时主动询问澄清。

#### 呈现格式

**意图确认**：
```

```

```

**进度提示**：
```
当前进度：X/Y

```

#### 适用场景

- 用户需求不够明确时的澄清

### 2.9 格式标准化

#### 目的

当AI输出内容时，遵循统一的格式标准，使人类能够：
- 预期一致的文档结构和呈现方式
- 快速定位所需信息
- 在不同输出之间保持理解的一致性

#### 描述

当AI输出内容时，应遵循统一的格式标准，包括标题层级、列表格式、术语使用等。

#### 呈现格式

**标题层级**：
- 一级标题：# 标题
- 二级标题：## 标题
- 三级标题：### 标题
- 四级标题：#### 标题
- 层级不宜超过4级

**列表格式**：
- 无序列表：使用"-"或"*"作为标记
- 有序列表：使用"1."、"2."、"3."作为标记
- 列表项内容不宜过长

**代码块**：
- 使用三个反引号标注代码块
- 指定语言类型
- 较长的代码行适当换行

**术语使用**：
- 同一概念全程使用相同术语
- 首次出现时提供定义或解释
- 避免同一概念的多种说法混用

#### 示例

##### 正确示例

```
# 用户登录功能设计

## 功能概述
[一段话介绍功能]

## 功能需求

### 1. 登录方式
- 用户名密码登录
- 手机验证码登录
- 第三方登录（可选）

### 2. 安全要求
1. 密码加密存储
2. 登录失败次数限制
3. 会话超时机制

## 数据模型

```python
class User:
    id: int
    username: str
    password_hash: str
    created_at: datetime
```

## API设计

| 方法 | 路径 | 说明 |
|:----:|:----:|:-----|
| POST | /api/login | 用户登录 |
| POST | /api/logout | 用户登出 |
| GET | /api/user | 获取用户信息 |
```

##### 错误示例（呈现层面的错误）

```
# 用户登录
## 1.功能需求
###（一）登录方式
用户可以...也可以...（格式不统一）
或者：
1.登录方式
a)用户名密码
b)验证码
2)安全要求（编号混乱）
```

#### 适用场景

- 所有正式输出
- 文档和报告编写
- 代码和API说明
- 多段落内容组织

#### 与其他模式的关系

- 与"层次化解释"模式配合：层级结构应遵循格式标准
- 与"结构化输出格式"模式配合：表格和列表格式应统一

### 3.1 置信度标注

#### 目的

当AI输出包含不确定性信息时，明确标注置信程度，使人类能够：
- 准确评估信息的可信赖程度
- 区分事实性内容和推测性内容
- 知道在哪些方面需要额外验证

#### 描述

当AI输出信息时，应使用统一的置信度标注系统，区分事实性内容和推测性内容，说明置信度判断的依据。

#### 呈现格式

**置信度等级**：
- **确定**（高置信度）：基于可靠来源的确定事实
- **推断**（中等置信度）：基于逻辑或证据的合理推断
- **推测**（低置信度）：可能性较高但缺乏充分证据的猜测
- **不确定**：AI不确定或不知道的内容

**标注方式**：
- 在相关陈述后使用括号标注，如"（确定）"
- 对于推断性内容，提供推断依据
- 对于推测性内容，说明推测的假设前提

#### 示例

##### 正确示例

```
根据官方数据，2023年中国GDP增长率为5.2%（确定，基于国家统计局数据）。

这表明经济运行整体回升向好（推断，基于GDP增长和多项经济指标）。

2024年预计经济增速可能在5%左右（推测，基于现有政策取向和历史规律，但存在不确定性）。

至于某个政策的具体影响，我目前没有足够信息来判断（不确定）。
```

##### 错误示例（呈现层面的错误）

```
2023年GDP增长5.2%（没有任何标注）
和
2024年可能会增长5%（说的跟确定事实一样）
以及
这个我不太确定但我觉得应该是这样的（没有明确标注为不确定）
```

#### 适用场景

- 包含事实性陈述的内容
- 预测性和推断性分析
- 需要区分确定性和可能性的场景
- 帮助用户评估信息可靠性

#### 与其他模式的关系

- 与"边界标注"模式配合：标注信息的适用范围
- 与"溯源标注"模式配合：提供信息来源
- 与"知识边界说明"模式配合：明确区分已知和未知

### 3.2 知识边界说明

#### 目的

当AI遇到超出知识范围的问题时，明确表示其知识边界，使人类能够：
- 知道哪些答案是AI确实知道的
- 理解AI能力的局限性
- 找到获取正确答案的途径

#### 描述

当AI遇到超出知识范围的问题时，应明确表示不知道，并说明其知识的时间边界和领域边界。

#### 呈现格式

**明确承认**：
```
关于[具体问题]，我目前没有足够信息来回答。

原因：
- 我的知识有截止日期，相关更新发生在截止日期之后
- 这不属于我训练数据覆盖的领域
- 这个问题需要专业领域的最新信息
```

**提供替代**：
```
虽然我不能直接回答这个问题，但您可以：
1. 查阅[相关资源/官方文档]
2. 咨询[专业人士]
3. 在[相关平台]搜索最新信息
```

#### 示例

##### 正确示例

```
问：2024年6月发布的最新手机芯片性能如何？

答：
关于2024年6月发布的最新手机芯片，我目前没有相关信息。

原因：
- 我的知识截止到2024年4月，无法获取之后发布的产品信息
- 芯片性能评测需要最新的测试数据

建议：
1. 查看芯片厂商官网或发布会信息
2. 参考科技媒体的评测文章
3. 关注专业测评机构的跑分数据
```

##### 错误示例（呈现层面的错误）

```
问：2024年6月发布的最新手机芯片性能如何？

答：
这个芯片性能挺好的，应该比上一代强不少，我觉得...（假装知道并乱说）
或者
哦你说这个啊我知道...（模糊应付过去）
```

#### 适用场景

- 涉及最新信息的问题
- 专业领域的细分问题
- 需要实时数据的问题
- 明确超出AI能力范围的问题

#### 与其他模式的关系

- 与"置信度标注"模式配合：不确定的内容应标注置信度
- 与"边界标注"模式配合：说明知识的适用范围
- 与"溯源标注"模式配合：提供可验证的信息来源

### 3.3 溯源标注

#### 目的

当AI输出包含事实性断言时，提供信息来源，使人类能够：
- 验证信息的准确性
- 追溯信息的来源
- 在需要时获取更多详细信息

#### 描述

当AI输出包含事实性断言时，应自动提供信息来源和引用，支持用户验证。

#### 呈现格式

**引用格式**：
- 使用标准的引用标记
- 提供可访问的链接或参考资料
- 区分直接引用和转述
- 标注信息的时间（如果是时间敏感的）

**引用示例**：
```
[1] 来源：XXX机构/网站
[2] 参考：XXX文献/报告
```

#### 示例

##### 正确示例

```
根据中国国家统计局发布的数据，2023年中国GDP同比增长5.2%[1]。

这一增速高于年初制定的5%左右的预期目标[2]。

分析认为，这一增长主要得益于[原因分析]...

---
参考来源：
[1] 国家统计局官网 - 2023年国民经济和社会发展统计公报
[2] 政府工作报告（2023年）
```

##### 错误示例（呈现层面的错误）

```
2023年中国GDP增长5.2%，这个数据是真的，我可以保证...（没有提供来源）
或者
根据某个报道/据说/好像...（来源模糊）
```

#### 适用场景

- 包含具体数据和统计信息的内容
- 引用事实、概念、定义的说明
- 需要权威性支撑的专业内容
- 用户可能需要验证的场景

#### 与其他模式的关系

- 与"置信度标注"模式配合：有来源的信息可以标注更高置信度
- 与"边界标注"模式配合：说明信息的适用范围
- 与"知识边界说明"模式配合：对于无法提供溯源的内容说明原因

### 3.4 一致性矛盾标注

#### 目的

当AI发现输出中存在潜在矛盾时，主动标注这些矛盾，使人类能够：
- 注意到可能的问题
- 有效验证信息的准确性
- 做出更准确的判断

#### 描述

当AI发现输出中存在潜在矛盾时，应主动标注这些矛盾，供人类判断和验证。

#### 呈现格式

**矛盾标注格式**：
```
⚠️ 潜在矛盾提醒：
- 矛盾点A：...
- 矛盾点B：...

建议：用户可自行验证以上内容
```

#### 示例

##### 正确示例

```
关于这个项目的成本和收益，我的分析如下：

## 成本分析
- 初期投资：100万元
- 运营成本：每年20万元

## 收益分析
- 年收入：25万元

⚠️ 潜在矛盾提醒：
- 按此计算，投资回收期约为100/(25-20)=20年
- 但前述分析提到"两年内可收回投资"

建议：请核实关键假设（如收入预测、运营成本）是否一致。

---

另一处：
关于该技术的应用范围：
- 应用范围1：xxx
- 应用范围2：yyy

⚠️ 潜在矛盾提醒：
- 前一个问题中您问的是"不能应用在哪些场景"，我回答的是xxx和yyy
- 这里的"应用范围"指的是"可以应用在"
- 请问您想了解的是"适用"还是"不适用"的场景？
```

##### 错误示例（呈现层面的错误）

```
前面说了A，后面说了B，这两者是矛盾的但我假装没看见，不做任何提示...
或者
我发现矛盾了但我不说，让用户自己去发现...
```

#### 适用场景

- 多部分内容的内部一致性检查
- 前后回答的对比和验证
- 复杂分析中多个因素的关系说明
- 用户指出矛盾后的回应

#### 与其他模式的关系

- 与"置信度标注"模式配合：矛盾内容可能涉及置信度问题
- 与"推理过程呈现"模式配合：推理过程中的矛盾应明确标注

### 3.5 可操作错误消息

#### 目的

当AI遇到错误或无法完成请求时，提供清晰、可操作的错误信息，使人类能够：
- 理解发生了什么问题
- 知道如何解决问题
- 有效继续或调整任务

#### 描述

当AI遇到错误或无法完成请求时，应使用用户能理解的语言，明确说明问题，并提供解决建议。

#### 呈现格式

**错误消息结构**：
```
❌ [错误类型]

问题描述：[用通俗语言说明发生了什么]

可能原因：
1. [原因1]
2. [原因2]

建议操作：
- [针对原因1的解决方法]
- [针对原因2的解决方法]

如需进一步帮助，请告诉我您的具体情况。
```

#### 示例

##### 正确示例

```
❌ 无法完成您的请求

问题描述：我无法直接访问您提到的内部系统或获取其中的数据。

可能原因：
1. 这是一个需要登录凭证的内部系统
2. 该系统没有对外开放的API接口
3. 我的访问权限不足以获取相关信息

建议操作：
- 如果您需要我分析这些数据，请将相关数据内容复制粘贴到对话中
- 如果您需要了解如何开放API接口，建议咨询系统管理员
- 如果您需要我生成测试数据，请告诉我数据格式和字段要求

请问您可以提供哪些信息来帮助我更好地协助您？
```

##### 错误示例（呈现层面的错误）

```
错误：Error 403
或者
抱歉，我做不到
或者
系统错误，请稍后重试（但用户不知道什么原因，也不知道能做什么）
```

#### 适用场景

- 无法满足用户请求的情况
- 输入信息不足或格式错误
- 遇到技术或权限限制
- 需要用户提供更多信息

#### 与其他模式的关系

- 与"边界标注"模式配合：说明能力边界
- 与"反馈明确表达"模式配合：明确表达系统状态
- 与"知识边界说明"模式配合：区分知识边界和错误情况

## 三、应用指南

### 3.1 模式组合建议

**场景：复杂问题解答**
推荐组合：
- 推理过程呈现 + 置信度标注 + 层次化解释
- 适用于：需要展示分析过程的专业问题

**场景：信息查询和事实说明**
推荐组合：
- 溯源标注 + 置信度标注 + 边界标注
- 适用于：提供需要验证的事实性信息

**场景：任务执行和操作指导**
推荐组合：
- 示例规范化呈现 + 结构化输出格式 + 格式标准化
- 适用于：需要明确输出格式的任务

**场景：交互式对话和咨询**
推荐组合：
- 反馈明确表达 + 角色一致呈现 + 可操作错误消息
- 适用于：模拟角色或提供专业咨询

### 3.2 效果评估

**评估维度**：

1. **可理解性**
   - 用户是否能快速理解AI输出的核心要点
   - 信息的组织结构是否清晰
   - 是否有明确的层级和逻辑

2. **可验证性**
   - 事实性信息是否有来源标注
   - 不确定性内容是否标注了置信度
   - 潜在矛盾是否被主动提示

3. **可用性**
   - 输出格式是否方便用户使用
   - 示例是否具有可模仿性
   - 错误信息是否具有可操作性

4. **一致性**
   - 格式是否在所有输出中保持一致
   - 角色呈现是否前后一致
   - 风格是否统一
