# plan-runner 合并回主干命令设计

## 背景

全局 `plan-runner` extension 当前已经支持：

- `/run-plan`
- `/run-status`
- `/run-summary`
- `/run-promote-docs`
- `/run-stop`

但缺少一个“将任务分支安全合并回主干”的收尾命令。当前 `run-plan` 会为任务创建 task branch，并在 `RunState` 中记录：

- `branchName`
- `repos`
- 每个 repo 的 `previousBranch`

这些信息已经足以支持“优先按活动 run 的元数据进行合并回主干”。

同时，用户也可能在没有激活 run 的情况下执行合并命令，因此还需要支持基于当前本地分支上下文自动识别多 repo 项目的同名任务分支，并完成合并回主干。

---

## 目标

新增一个全局命令：

- `/run-merge`

满足以下行为：

1. **优先基于活动 run 合并回主干**
   - 若存在激活的 plan run，则使用该 run 的：
     - `branchName` 作为源分支
     - `repos` 作为作用 repo 集合
     - 每个 repo 的 `previousBranch` 作为目标主干

2. **无活动 run 时支持自动推断**
   - 若没有激活 run，则根据当前本地分支上下文自动推断：
     - 任务分支名
     - 需要一起处理的 repo 集合
     - 每个 repo 的目标主干

3. **默认采用 squash merge**
   - 默认以压缩合并的形式将任务分支内容整理后落到主干

4. **支持在改动明显可分时拆成多个原子提交**
   - 当改动范围较大、且明显可拆为多个关注点时，允许生成多个原子提交
   - 拆分以当前内容语义为主，原提交历史只作辅助参考

5. **冲突且无法简单判断时必须先问用户**
   - 若合并预演出现冲突，且 extension 无法简单判断如何安全合并，则提示用户：
     - 是否先在任务分支上合并主干并验证
     - 然后再回来执行 `/run-merge`

6. **没有活动 run 时，以分叉点之后的提交信息作为提交说明参考**
   - 当无活动 run 时，提交说明以 `merge-base(target, source)..source` 范围内的 commit subjects 为主要参考来源

---

## 非目标

本次设计不包含：

- 改变 `run-plan` 的整体执行模型
- 将“合并回主干”改造成新的子 `pi` 执行流程
- 自动进行复杂语义级冲突解决
- 增加跨 repo 的真正事务性提交机制
- 递归扫描任意深度目录查找 repo

---

## 用户工作前提

执行 `/run-merge` 时，默认假定：

1. 用户当前处于“待合并任务分支”的本地上下文中
2. 当前目录 `cwd` **可以不是 git repo**
3. 在多 repo 项目中，如果 `cwd` 不是 repo，则只在其**直接子目录**中寻找 repo
4. 用于合并预演的 `worktree` 由命令内部临时创建与清理，**不是要求用户预先准备的 worktree**

---

## 方案对比

### 方案 1：直接在当前 worktree 上切主干并 merge

做法：

- 直接在当前工作区切到目标主干
- 执行 squash merge 或 cherry-pick
- 发生冲突时在当前工作区处理

优点：

- 实现简单
- 改动最少

缺点：

- 冲突时容易污染用户当前工作区
- 多 repo 场景下恢复成本高
- 对失败回滚与状态清理不够稳健

结论：不采用。

### 方案 2：使用临时 worktree 做“预演 + 落地”

做法：

- 先解析 repo 集合、源分支、目标主干
- 在每个 repo 上创建临时 worktree，基于目标主干做预演合并
- 只有在预演通过后，才在临时 worktree 中生成最终 1 个或多个提交
- 最后再把结果落到目标主干

优点：

- 预演失败不会污染用户当前目录
- 更容易在冲突前置暴露问题
- 更适合 plan-runner 的多 repo 场景

缺点：

- 实现比直接 merge 略复杂
- 需要明确 worktree 清理逻辑

结论：**采用本方案**。

### 方案 3：将提交拆分与 message 生成交给独立子 `pi`

做法：

- extension 只负责 git 护栏和上下文解析
- 关注点拆分与提交信息生成交给子 `pi` 决定

优点：

- 灵活性高

缺点：

- 超出当前最小必要范围
- 提高复杂度与验证成本
- 把一个偏确定性的 git 命令变成更重的 AI 编排

结论：本次不采用。

## 设计总览

新增命令：

- `/run-merge`

设计核心：

1. **优先使用 active run 元数据**恢复“这次计划涉及哪些 repo、各自从哪个主干分出”
2. **无 active run 时按同名任务分支识别 repo 组**
3. **先在临时 worktree 里预演合并**，避免污染用户当前目录
4. **默认 squash**，仅在改动明显可分时才拆成多个原子提交
5. **碰到冲突且不能简单判断时，先询问用户**是否改走“先在任务分支上合并主干并验证”的人工路径

---

## 命令接口

### 新命令

- `/run-merge`

建议描述：

- `Merge the current task branch back to trunk (prefers active run metadata, otherwise infers from same-named branches)`

### 输出风格

命令应在 UI 中明确展示：

- 采用的源分支名
- 目标主干分支名（每个 repo 各自显示）
- 参与处理的 repo 列表
- 是否使用单 squash commit 或多个原子提交
- 结果摘要（成功 / 用户确认中止 / 冲突阻塞）

---

## 作用范围解析

### 情况 A：存在 active run

当存在激活 run 时，作用范围完全以 `RunState` 为准：

- **源分支**：`activeRun.branchName`
- **repo 集合**：`activeRun.repos`
- **目标主干**：每个 repo 的 `previousBranch`

这保证了：

- 单 repo 任务可回到启动 run 前所在主干
- 多 repo 任务可回到各自启动 run 前记录的主干

这是 `/run-merge` 的优先路径。

### 情况 B：不存在 active run

当没有激活 run 时，按当前上下文自动推断。

#### B1. `cwd` 本身是 git repo

- 读取当前 repo 的当前分支名，作为候选任务分支名
- 找到工作区中所有处于该同名分支的 repo
- 对每个 repo 自动识别默认主干

#### B2. `cwd` 本身不是 git repo

- **不能直接终止**
- 只扫描 `cwd` 的**直接子目录**
- 找出其中是 git repo 的目录
- 读取这些 repo 当前所在分支

如果这些 repo 的当前分支不一致，则：

1. 先排除各 repo 的默认主干分支
2. 在剩余分支中找出**出现次数最多**的分支名，作为候选任务分支
3. **但不直接继续**，先向用户确认

示例确认文案：

> 检测到多个 repo 当前分支不一致，准备以 `pi/20260330-foo` 作为任务分支，并处理这些同名分支 repo：A、B、C。是否继续？

用户确认后，命令再继续收集并处理所有处于该同名任务分支的 repo。

---

## 主干分支识别规则

### 有 active run 时

目标主干直接取 `previousBranch`，不再动态猜测。

### 无 active run 时

自动识别默认主干，优先级为：

1. `origin/HEAD` 指向的默认分支
2. 本地存在的 `main`
3. 本地存在的 `master`
4. 本地存在的 `trunk`

若仍无法确定，则停止并报告。

---

## 前置校验

以下情况应在真正开始预演前直接停止并报告：

- 当前上下文下无法找到任何 repo
- 任一候选 repo 工作区不干净
- 任一候选 repo 处于 detached HEAD
- 无 active run 且最终无法确定候选任务分支名
- 任一 repo 找不到明确的目标主干
- 用户当前已经位于主干且无明确 task branch 上下文

这些校验的目标是：在进入 worktree 预演之前，尽可能把不确定性前置暴露。

---

## 合并策略

### 总体流程

对每个 repo 按下列顺序准备：

1. 解析源分支与目标主干
2. 计算分叉点：`merge-base(target, source)`
3. 创建基于目标主干的临时 worktree
4. 在临时 worktree 中预演合并
5. 所有 repo 预演都通过后，再进行最终提交生成与落地

多 repo 模式下，采用“两阶段”思路：

- **阶段 1：全部 repo 完成前置检查与预演**
- **阶段 2：全部通过后再统一落地**

这样可以避免一部分 repo 已经合并回主干、另一部分 repo 却因冲突卡住。

### 默认行为：squash merge

默认采用：

- 单个 squash commit

原因：

- 与需求中“默认压缩合并”一致
- 能整理由任务分支上产生的中间噪声提交
- 使 task branch -> trunk 的落地主线更清晰

---

## 冲突处理

### 预演无冲突

若临时 worktree 中的合并预演无冲突，则继续进行：

- 提交信息生成
- 单提交或多提交拆分
- 最终落地

### 预演出现冲突

若预演进入冲突态：

- extension **不做自动语义级冲突解决**
- 也不在用户当前目录中留下冲突现场
- 先判断是否属于“可以简单判断的机械合并”

若**无法简单判断如何安全合并**，则必须询问用户：

> 合并到主干时出现冲突，当前无法安全判断如何自动合并。是否先在任务分支上合并主干并完成验证，再回来执行 `/run-merge`？

这对应的策略是：

- **能简单、安全判断**：继续
- **不能简单判断**：停止自动落地，转为用户决策

本次设计中，为了保持行为保守可靠，默认实现应倾向于：

- 冲突即认为“无法简单判断”，优先询问用户

而不是尝试复杂自动合并。

---

## 提交拆分策略

### 默认：单 squash commit

除非满足“明显可分”的条件，否则始终保持单 squash commit。

### 何时允许拆成多个原子提交

仅当同时满足以下条件时，才允许拆分为 2~4 个提交：

1. 改动量较大
2. 内容明显分为多个关注点
3. 各关注点文件边界比较清晰
4. 关注点之间重叠较少
5. 源分支历史能对这些关注点划分提供辅助证据

### 不拆分的情况

以下任一情况成立，则保持单 squash commit：

- 改动高度交织
- 关注点划分不稳定
- 原提交历史噪声较多
- 为了拆分而拆分会明显增加出错风险

### 拆分原则

拆分策略遵循：

- **优先按当前内容重新组织**
- **必要时参考原提交历史**
- **不盲信原始提交序列**

也就是说：

1. 先根据当前 diff、目录边界、文件集合判断是否存在多个自然关注点
2. 再参考源分支分叉点之后的 commit subjects 与触及路径
3. 只有当“内容分组”和“历史边界”大致一致时，才真正生成多个提交
4. 否则回退到一个 squash commit

## 提交信息生成规则

### 单个 squash commit

#### 有 active run 时

优先级如下：

1. 计划标题（如 `# [Feature Name] Implementation Plan`）
2. 计划中的 `Goal`
3. 计划文件名
4. 回退到通用 message

生成后的提交信息应尽量符合 Conventional Commits 风格，例如：

- `feat: add plan runner branch merge command`
- `fix: improve plan runner merge-back flow`
- `docs: update plan runner merge workflow`

若无法可靠判断类型，则保守回退为：

- `chore: merge <branch> back to <trunk>`

#### 无 active run 时

提交信息应以分叉点之后的提交信息作为主要参考来源：

- 范围：`merge-base(target, source)..source`
- 收集 commit subjects
- 若主题单一，则直接复用或轻微整理
- 若主题较多，则提炼总主题
- 若最终拆成多个提交，则每个分组分别参考对应 subjects 生成 message

### 多个原子提交

若最终拆成多个提交，则要求：

- 每个提交都只承载一个清晰关注点
- message 同时参考：
  - 当前分组的文件内容 / 目录边界
  - 对应历史 commit subjects
- 只要某个分组的 message 不够稳定，就退回单 squash

---

## 多 repo 一致性策略

本设计不引入真正的跨 repo 事务机制，但采取如下流程约束：

1. 先对所有 repo 进行前置检查
2. 再对所有 repo 进行 worktree 预演
3. 任一 repo 在预演阶段失败，则整体停止，不对其他 repo 主干落地
4. 只有全部 repo 都通过预演后，才开始最终提交与落地

这可以尽量避免“同名任务分支的一部分 repo 已落地，另一部分尚未落地”的不一致状态。

## 状态与工件记录

### 有 active run 时

复用现有 `.pi/runs/<runId>/` 目录，但不新建新的 run 生命周期。

建议新增 merge 结果工件，例如：

- `.pi/runs/<runId>/merge-result.json`
- 或 `.pi/runs/<runId>/merge-summary.md`

记录内容包括：

- merge 执行时间
- 参与 repo 列表
- 每个 repo 的源分支与目标主干
- 是否采用 squash
- 是否拆分为多个提交
- 最终 commit ids
- 是否因为冲突或用户取消而中止

### 无 active run 时

不强制写入 `.pi/runs/...`，只需在 UI / command output 中清晰汇总：

- 识别结果
- 用户确认信息
- 最终 merge 结果

## 用户交互点

为了避免命令过程过于冗长，仅保留两个必须的确认点：

### 确认点 1：无 active run 且分支推断不唯一

当 `cwd` 不是 git repo，且直接子目录中 repo 当前分支不一致时：

- 选择“出现次数最多的非主干分支”作为候选任务分支
- 但必须先向用户展示：
  - 检测到的 repo 分支分布
  - 准备采用的任务分支
  - 将要处理的 repo
  - 对应目标主干
- 用户确认后才继续

### 确认点 2：预演冲突且无法简单判断

当出现冲突时：

- 告知哪个 repo 冲突
- 告知源分支 / 目标主干
- 明确当前只是临时 worktree 预演，未污染当前目录
- 询问是否改为“先在任务分支上合并主干并验证”的路径

## 错误处理与清理

以下情况直接停止并报告，不做隐式修复：

- 任一 repo 工作区不干净
- 任一 repo 处于 detached HEAD
- 找不到明确的源分支或目标主干
- 预演冲突且无法简单判断
- 多 repo 中任一 repo 预演失败
- 尝试拆分原子提交时发现关注点重叠过重

同时必须保证：

- 预演失败不污染用户当前工作区
- 临时 worktree / 临时引用在成功或失败后都被清理
- 只有在全部 repo 前置检查通过后，才进入最终落地阶段

## 实现边界与取舍

### 为什么使用临时 worktree

因为它能在以下几点之间取得更好的平衡：

- 不污染用户当前目录
- 允许先发现冲突再决定是否继续
- 适配多 repo 场景
- 实现复杂度仍低于引入完整子进程编排

### 为什么默认倾向单 squash

因为：

- 这是用户明确要求的默认行为
- 它最稳定
- 它可以避免过度依赖任务分支中的噪声提交历史

### 为什么冲突时先问用户，而不是自动尝试解决

因为 extension 缺乏足够上下文去进行可靠的业务级冲突裁决。相比“猜着合并”，更安全的做法是：

- 在临时 worktree 中发现冲突
- 说明当前无法简单判断
- 交还给用户决定是否先在任务分支整合主干并验证

---

## 最终结论

采用“**优先 active run 元数据 + 无 run 时按同名任务分支自动识别 + 临时 worktree 预演 + 默认 squash + 冲突时用户确认**”的方案。

该方案能够满足以下核心需求：

- 从 plan-runner 的 task branch 安全合并回主干
- 支持多 repo 项目
- 在没有激活 run 时仍能基于当前上下文工作
- 避免污染用户当前工作区
- 在冲突且无法简单判断时先询问用户
- 默认压缩合并，并在改动明显可分时允许拆成多个原子提交
- 没有激活 run 时，使用分叉点之后的提交信息作为提交说明参考
