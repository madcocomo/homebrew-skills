# 独立 pi 进程编排工作流

把“局部阅读 + 局部分析 + 局部修改 + 局部验证 + 第一轮失败诊断”尽量放进独立 pi 进程，只把任务边界、验收标准和最终摘要留在主会话。

## 适用场景

优先在这些场景使用：
- 范围清晰、可独立验收的子任务
- 局部重构、定点调试、定向测试、模块审查
- 多个串行子任务，且每一步都有明确继续门槛
- 多个彼此独立、可以并行展开、最后统一汇总的子任务
- 希望主会话只保留主线目标，不被局部细节污染

不要为了极小的一步操作强行拆分；如果主会话几乎不用读代码、不用分析、也不用验证，直接在主会话完成通常更简单。

## 核心原则

1. 先判断子任务是否真的独立：边界、范围、验收标准必须清楚。
2. 主会话只负责编排，不负责子任务内部的大量细节分析。
3. 子进程尽量完成完整闭环，而不是只替主会话“代跑一条命令”。
4. 主会话默认只读取精简摘要，不读取完整事件流或大段日志。
5. 有明确串行门槛时，满足门槛后自动推进；只有失败、歧义、范围扩大时才回来确认。
6. 使用 `tmux new-session -e ...` 传递环境变量，避免在命令里堆叠大量 `export`。
7. 如果多个子任务之间不存在读写冲突、顺序依赖或共享临时状态，优先并行展开，再在主会话做一次集中汇总。

## 模式判断：规划模式 vs 执行模式

在真正启动独立 pi 进程之前，先判断用户当前要的是“编排设计”还是“立即执行”。这一步非常重要。

### 规划模式（默认优先）

如果用户主要在问这些内容，默认进入规划模式：
- 怎么组织流程
- 该串行还是并行
- 给我模板 / 方案 / 工作流
- 如何拆子任务
- 如何做 fan-in 汇总
- 先别执行，先设计一下
- 评审一下这套编排是否合理

规划模式下：
- **不要真的启动** `tmux`、独立 pi 进程、构建命令或测试命令
- 只输出：
  - 任务拆分
  - session 命名建议
  - task file / status / summary / log 结构
  - gate / batch / fan-in 规则
  - 主会话回收策略
- 如果需要示例，给出可执行模板，但不要替用户真正开跑

### 执行模式（必须有明确执行信号）

只有在用户明确表达这些意图时，才进入执行模式：
- 现在启动
- 直接执行
- 帮我开跑
- 继续做下去
- 按这个方案实际落地
- 已经给出足够的工作目录、范围、验证命令，并明确希望你开始执行

执行模式下才允许：
- 启动 `tmux` session
- 真的创建 task file 并调用 `pi`
- 运行构建、测试、验证命令
- 回收状态文件和摘要

### 歧义时的默认规则

如果用户同时提到了“流程设计”和“实际执行”，但没有明确说现在就开跑，**默认先走规划模式**。

宁可先给出一份清晰的编排方案，也不要因为技能被触发就自动开始拉起子进程。这个技能的目标是提升编排质量，而不是让所有相关请求都变成立即执行。

## 主会话职责

主会话只做这些事：
- 定义任务目标
- 定义工作目录
- 定义允许修改范围与禁止修改范围
- 定义验证命令与验收标准
- 定义串行依赖顺序
- 读取子任务摘要
- 判断是否进入下一步

主会话默认不要做这些事：
- 大量局部代码阅读
- 大量局部日志阅读
- 子任务内部第一轮排障
- 为了“省一步”而把本该下沉的分析重新拉回主会话

## 子进程职责

独立 pi 进程应尽量在自己的上下文里完成整个局部闭环：
- 阅读目标范围代码或文件
- 形成局部方案
- 修改文件
- 运行验证命令
- 对失败做第一轮诊断
- 输出压缩后的状态与摘要

目标不是“远程执行器”，而是“局部负责到底的子任务代理”。

## 何时启动独立 pi 进程

**只有在已经进入执行模式时**，才考虑启动独立 pi 进程。

满足以下大多数条件时，尽早启动，而不是先在主会话里做完整分析：
- 任务边界清楚
- 输入资料基本齐全
- 验收标准可描述
- 失败后也能在原边界内先做一轮诊断
- 完成后主会话只需要摘要即可继续

如果当前仍是规划模式，就只描述如何启动、如何拆分、如何回收结果，不要真的启动。
## 启动规则

始终遵循：
- 用 `tmux` 承载独立进程
- 用 `zsh -c 'source "$HOME/.zshrc" && ...'`
- 需要代理时，通过 `tmux new-session -e ...` 传递环境变量
- 尽量让子进程把结果写到文件，再由主会话读取摘要文件

### 推荐启动模板

```bash
tmux new-session -d \
  -e http_proxy="$http_proxy" \
  -e https_proxy="$https_proxy" \
  -e HTTP_PROXY="$HTTP_PROXY" \
  -e HTTPS_PROXY="$HTTPS_PROXY" \
  -e all_proxy="$all_proxy" \
  -e ALL_PROXY="$ALL_PROXY" \
  -e no_proxy="$no_proxy" \
  -e NO_PROXY="$NO_PROXY" \
  -s <session-name> \
  "cd <workdir> && zsh -c 'source \"$HOME/.zshrc\" && pi --mode json -p @<task-file> > <result-json> 2> <stderr-log>'"
```

### 命名建议

- session 名应体现任务目的，如 `pi-refactor-controller`、`pi-test-regression`、`pi-review-module-a`
- 任务文件、状态文件、摘要文件使用同一前缀，方便主会话收集

## 子任务输入规范

为每个子任务准备一个任务文件，例如 `/tmp/pi-task-1.md`。

需要实际可复用模板时：
- 串行子任务模板：`references/serial-task-template.md`
- 并行批次任务模板：`references/parallel-batch-template.md`
- 主会话 fan-in 汇总模板：`references/fan-in-summary-template.md`
- 场景示例与判定参考：`references/examples.md`

建议结构：

```md
任务：<一句话目标>

工作目录：<absolute-path>

范围：
- 允许读取/修改的文件或目录

禁止修改：
- 明确不允许动的部分

要求：
1. 先读后改
2. 保持最小改动
3. 在子进程内完成局部分析与验证
4. 失败时先自行诊断并在边界内重试一次（如合适）

验证：
- <command 1>
- <command 2>

最终输出请包含：
- 修改文件
- 核心改动
- 验证结果
- 风险与后续建议
- 状态文件与摘要文件路径
```

如果是串行任务，在任务文件里明确写清：
- 当前步骤序号
- 本步骤通过后允许自动进入的下一步
- 继续门槛是什么

如果是并行任务，在任务文件里明确写清：
- 本任务所属批次（batch）
- 与哪些任务互相独立
- 禁止修改的共享资源
- 完成后应输出到哪个汇总目录
- 是否需要在汇总阶段做去重、比对或合并

## 子任务输出规范

至少产出三类文件。

### 1. 状态文件
例如：`/tmp/pi-task-1.status.json`

```json
{
  "success": true,
  "summaryFile": "/tmp/pi-task-1.summary.md",
  "modifiedFiles": [
    "src/main/java/..."
  ],
  "verifications": [
    {
      "command": "mvn test ...",
      "success": true,
      "log": "/tmp/pi-task-1.test.log"
    }
  ]
}
```

### 2. 摘要文件
例如：`/tmp/pi-task-1.summary.md`

摘要应尽量短，只保留：
- 修改文件
- 核心改动
- 验证结果
- 风险或后续建议
- 是否建议进入下一步

不要把完整中间推理和长日志直接塞进摘要。

### 3. 详细日志
例如：
- `/tmp/pi-task-1.stderr.log`
- `/tmp/pi-task-1.full.json`
- `/tmp/pi-task-1.test.log`

这些文件是兜底材料。主会话默认不读，只有失败或摘要不足时才读取。

## 输出消费规则

主会话读取顺序应为：
1. `status.json`
2. `summary.md`
3. 仅在必要时再看 `stderr.log`、`full.json`、测试日志

如果只需要最终结论，不要直接消费完整事件流。

即使使用 `--mode json`，也仍然要求子进程额外输出适合主会话消费的 `status.json` 和 `summary.md`。

## 串行任务推进规则

如果用户已经明确给出一组串行独立任务，并且下一步继续条件明确，则按以下方式执行：

1. 主会话定义整条链路与每步门槛。
2. 子进程完成当前步骤并返回状态与摘要。
3. 若门槛满足，则主会话直接启动下一子任务。
4. 只有遇到下面情况才回来确认：
   - 前一步失败
   - 结果与验收标准不符
   - 下一步范围明显扩大
   - 风险明显上升
   - 用户要求逐步确认

## 并行任务编排规则

当多个子任务满足“相互独立、无顺序要求、无写冲突、可独立验收”时，优先并行而不是串行。

适合并行的典型场景：
- 分别审查多个互不相关的模块
- 对多个文件做相同类型的局部分析
- 同时跑多组独立测试或回归检查
- 为多个候选方案各自生成结果，再统一比较

不要并行的场景：
- 会修改同一批文件
- 共享同一临时目录或中间产物
- 必须依赖上一步输出才能继续
- 需要强一致的全局判断后才能落地修改

并行执行时按以下方式组织：

1. 主会话先切分任务，确保每个子任务边界互不重叠。
2. 为同一批次任务分配统一的 batch 标识，例如 `batch-a`。
3. 为每个子任务单独准备任务文件、状态文件、摘要文件和日志文件。
4. 用不同的 tmux session 同时启动多个独立 pi 进程。
5. 主会话等待所有任务完成后，再做一次 fan-in 汇总。
6. 汇总时优先读取每个任务的 `status.json` 与 `summary.md`，只在异常项上展开日志。

### 并行命名建议

同一批次建议使用统一前缀，例如：
- session：`pi-batch-a-task-1`、`pi-batch-a-task-2`
- task file：`/tmp/pi-batch-a-task-1.md`
- status：`/tmp/pi-batch-a-task-1.status.json`
- summary：`/tmp/pi-batch-a-task-1.summary.md`

### 并行启动模板

仅在执行模式下使用：

```bash
tmux new-session -d \
  -e http_proxy="$http_proxy" \
  -e https_proxy="$https_proxy" \
  -e HTTP_PROXY="$HTTP_PROXY" \
  -e HTTPS_PROXY="$HTTPS_PROXY" \
  -e all_proxy="$all_proxy" \
  -e ALL_PROXY="$ALL_PROXY" \
  -e no_proxy="$no_proxy" \
  -e NO_PROXY="$NO_PROXY" \
  -s pi-batch-a-task-1 \
  "cd <workdir> && zsh -c 'source \"$HOME/.zshrc\" && pi --mode json -p @/tmp/pi-batch-a-task-1.md > /tmp/pi-batch-a-task-1.full.json 2> /tmp/pi-batch-a-task-1.stderr.log'"

# 同批次其他任务使用不同 session / task file / output file 并行启动
```

### 并行汇总规则

主会话在汇总时只做这些事：
- 收集所有子任务的成功/失败状态
- 对比修改范围是否真的互不重叠
- 汇总关键发现、验证结果与风险
- 判断是否需要进入下一轮串行收敛步骤

如果并行批次的目标是“分析”而不是“落地修改”，可以让多个子进程都只产出建议，由主会话在汇总后统一决定下一步修改方案。

需要统一汇总格式时，使用：
- `references/fan-in-summary-template.md`

### 并行失败策略

并行批次中某个任务失败时，不要立即判定整批失效。按以下顺序处理：
- 先看该子任务是否能在原边界内自行修复并重试一次
- 若部分任务成功，先保留成功任务的摘要与产物
- 由主会话判断失败任务是否阻塞整批目标
- 只有当失败任务影响最终收敛时，才暂停后续步骤并升级处理

### 并发度建议

默认保持保守并发，避免同时启动过多会话造成资源争用。
经验上应优先选择“少量高质量并行”，而不是把所有碎任务一次性铺满。
如果任务会消耗大量 CPU、内存、网络或构建缓存，应降低并发度。

## 串并行混合策略

很多真实任务不是纯串行或纯并行，而是：
- 先串行定义边界
- 中间并行展开若干独立子任务
- 最后再串行汇总、决策、落地

推荐的模式是：
1. 串行：定义目标、拆分任务、设定验收标准
2. 并行：对子模块/子问题分别执行阅读、分析、验证
3. 串行：汇总结果，决定统一修改或下一步计划
4. 必要时再开启下一轮并行批次

## 失败处理规则

子进程失败时，不要立刻把所有问题抛回主会话。先在子进程内部完成第一轮诊断：
- 查看本任务的 stderr、测试日志、结果文件
- 判断失败属于环境问题、构建问题、测试失败、代码错误，还是网络/代理问题
- 若能在原任务边界内修复，则优先修复并重试一次
- 如果仍失败，再输出精简失败报告给主会话

失败摘要至少应包含：
- 失败原因
- 已尝试修复
- 当前阻塞点
- 建议下一步

## 决策启发

如果你发现主会话已经开始做这些事，就说明拆分还不够彻底：
- 反复阅读局部实现细节
- 手动分析大量测试输出
- 帮子进程做第一轮错误归因
- 把多个独立步骤混在一个大上下文里处理

此时应回到“边界 + 验收 + 摘要”思路，重新切分子任务。

## 最终汇报模板

向主会话回传时，优先使用类似结构：

```md
# 子任务摘要
- 任务：<task name>
- 结果：成功 / 失败
- 修改文件：...
- 核心改动：...
- 验证结果：...
- 风险：...
- 下一步建议：...
```

## 快速检查清单

在启动前检查：
- [ ] 子任务范围清楚
- [ ] 验收标准清楚
- [ ] 工作目录清楚
- [ ] 会话名清楚
- [ ] 输出文件路径清楚

在回收结果时检查：
- [ ] 先读 `status.json`
- [ ] 再读 `summary.md`
- [ ] 失败时才展开详细日志
- [ ] 满足门槛则自动推进下一步

## 一句话目标

让主会话只保留主线控制权，让独立 pi 进程承担局部执行责任，从而以更低的上下文成本模拟稳定、可回收、可串行推进的 subAgent 工作流。
