# 独立 pi 进程编排示例

当需要判断应该串行、并行，还是串并行混合时，优先参考这些例子。

## 示例 1：纯串行

### 场景
需要先修改 `acorn`，验证通过后再去 `avia-base` 适配调用方。

### 为什么必须串行
- `avia-base` 依赖 `acorn` 的新产物
- 前一步不通过，后一步继续没有意义
- AH 工作区还有“修改 acorn 后必须运行 `mvn install`”的明确门槛

### 推荐流程
1. 子任务 A：修改 `acorn`
2. 子任务 A：运行 `mvn install`
3. 主会话读取 A 的摘要
4. 若通过，再启动子任务 B：修改 `avia-base`
5. 子任务 B：运行验证
6. 主会话汇总结果

### 不推荐做法
- 同时修改 `acorn` 和 `avia-base`
- 在 `acorn` 未验证前就让 `avia-base` 开始适配

## 示例 2：纯并行

### 场景
要分别审查 3 个互不相关的 controller，目标是找出重复异常处理并各自给出建议。

### 为什么适合并行
- 各自负责不同文件
- 彼此没有顺序依赖
- 可以独立输出摘要
- 主会话最后统一比较和汇总即可

### 推荐流程
1. 主会话切分成 3 个子任务
2. 为每个子任务生成独立 task file
3. 用 3 个 tmux session 并行启动 3 个 pi 进程
4. 等待全部结束
5. 主会话读取 3 份 `status.json` + `summary.md`
6. 做 fan-in 汇总，决定是否进入统一重构阶段

### 关键约束
- 不要让 3 个任务同时改公共基类
- 如果任一任务发现需要抽公共层，先在摘要里上报，不要自行扩大范围

## 示例 3：串并行混合

### 场景
你要完成一次较复杂的 API 改造：
- 先明确接口边界与验收标准
- 再分别审查 controller、service、BDD tests
- 最后统一决定最小改动方案并落地

### 推荐流程

#### 第一阶段：串行定义边界
主会话先明确：
- API 行为目标
- 可修改范围
- 验证方式
- 哪些部分可以独立分析

#### 第二阶段：并行展开分析
并行启动 3 个子任务：
- 任务 A：审查 controller 层
- 任务 B：审查 service 层
- 任务 C：审查测试覆盖与缺口

要求这 3 个任务先以“分析 + 建议”为主，不急着各自落地修改。

#### 第三阶段：串行收敛
主会话统一读取 3 份摘要后：
- 判断是否存在冲突
- 确定统一修改方案
- 选择一个新的子任务去集中落地修改
- 最后再由一个子任务执行验证并输出摘要

### 为什么这样更稳
- 分析阶段并行，节省时间
- 落地阶段串行，避免互相踩文件
- 主会话只在关键决策点介入

## 示例 4：不应并行的反例

### 场景
两个子任务都想修改同一个公共工具类和同一组测试文件。

### 为什么不能并行
- 修改范围重叠
- 极易产生覆盖、冲突或重复劳动
- 主会话最终很难判断哪一个结果是正确收敛版本

### 正确做法
- 先并行做只读分析
- 汇总后，启动单个串行落地任务去统一修改

---

## 经验规则

### 适合并行
- 读多写少
- 文件边界清晰
- 结果可独立验收
- 汇总主要是收集观点、风险、测试结果

### 适合串行
- 强依赖前一步结果
- 会改同一批文件
- 需要统一全局判断后才能落地
- 验收门槛严格串联

### 适合串并行混合
- 复杂任务可以分阶段
- 前期以分析为主，后期以统一落地为主
- 既想节省时间，又想减少合并冲突
