# 信息呈现模式 Skill

## 核心原则

本skill指导AI以人类友好的方式呈现信息。**核心目标是选择合适的可视化形式来组织信息**，而非使用特定的图形工具。可视化形式应该根据信息内容本身来选择，可以是：
- 结构化文本（标题层级、列表、编号）
- 表格（对比、矩阵）
- 图形化表示（流程图、状态图、时序图）
- 具体实例
- 场景化描述
- 可执行的验证代码

## 何时触发

在以下场景中，你应该激活此skill：

1. **用户要求解释代码、文档、概念时**
   - 用户让你解释某个代码文件
   - 用户让你解释某个概念或原理
   - 用户让你说明某个文档的内容

2. **AI完成工作进行总结时**
   - 代码生成完成后需要说明实现逻辑
   - 分析完成后需要呈现结论
   - 任务完成后需要报告结果

3. **AI需要向人提出澄清问题时**
   - 发现需求不明确时
   - 需要确认用户意图时
   - 发现潜在问题时

4. **当需要生成说明文档时**
   - 生成README文件
   - 生成API文档
   - 生成使用说明

## 快速决策：何时画图

**原则：复杂内容用图，简单内容用字。一句话能说清的不画。**

| 场景 | 图类型 |
|------|--------|
| 有调用顺序 | 时序图 |
| 有层级结构 | 架构图 |
| 有状态变化 | 状态机 |
| 有数据流转 | 数据流图 |
| 比较方案 | 对比图 |
| ≥3 个步骤 | 流程图 |
| 一句话能说清 | 不画 |

## 模式选择指南

根据你要呈现的信息类型，选择合适的可视化模式：

### 流程性信息（步骤、分支、循环）

适用模式：
- **流程图模式**：展示步骤顺序和分支条件
- **泳道图模式**：展示多角色协作的流程
- **序列图模式**：展示组件间的交互时序
- **状态机模式**：展示状态流转

### 规则性信息（条件、结果）

适用模式：
- **判定矩阵模式**：展示多维度条件与结果的对应关系
- **极端示例模式**：通过边界值测试暴露问题
- **反向验证模式**：验证反面情况是否成立

### 概念性信息（定义、关系）

适用模式：
- **实例化模式**：用具体实例说明抽象概念
- **场景化描述模式**：构建具体场景来串联知识点
- **可执行验证模式**：通过实际执行验证正确性

## 模式详解

### 1. 流程图模式

将过程性、步骤性的信息以图形化方式呈现。

**适用场景**：
- 需求处理流程
- 数据处理流水线
- 操作步骤说明

**实施要点**：
1. 识别信息中的步骤序列
2. 确定每个步骤的输入和输出
3. 识别条件分支和并行分支
4. 使用适当的可视化形式（ASCII图形、Mermaid、文本流程图等）

### 2. 泳道图模式

将多角色协作的流程以泳道方式呈现，展示每个角色的职责边界。

**适用场景**：
- 多角色交互的业务流程
- 系统架构设计
- 跨系统协作流程

### 3. 序列图模式

将系统交互和消息传递以时序方式呈现。

**适用场景**：
- API调用说明
- 组件交互设计
- 消息队列流程

### 4. 状态机模式

将系统状态变化和事件响应以状态机方式呈现。

**适用场景**：
- 业务状态流转规则
- 工作流设计
- 生命周期管理

### 5. 判定矩阵模式

将多维度条件与结果的对应关系以矩阵形式呈现。

**适用场景**：
- 多条件业务规则
- 权限控制规则
- 定价规则

**实施要点**：
1. 识别所有条件及其取值
2. 生成所有条件组合
3. 为每种组合标注结果
4. 标识不可能或无效的组合

### 6. 实例化模式

通过具体实例说明抽象概念，消除理解歧义。

**适用场景**：
- 概念定义
- 规则说明
- 数据结构说明

**实施要点**：
1. 识别抽象概念和术语
2. 提供具体实例
3. 标注边界情况

### 7. 极端示例模式

通过边界值和极限情况暴露潜在问题。

**适用场景**：
- 计算公式
- 验证规则
- 数量限制

### 8. 反向验证模式

通过验证命题的反面发现遗漏和错误。

**适用场景**：
- 规则描述
- 约束条件
- 验证逻辑

### 9. 场景化描述模式

通过具体场景将零散知识点串联起来。

**适用场景**：
- 功能说明
- 业务流程
- 业务规则

### 10. 可执行验证模式

当信息可以被执行时，直接运行验证。

**核心原则**：尽可能降低人类的验证成本，让AI承担验证的执行工作。

**实施要点**：
1. 优先AI直接执行验证
2. 呈现预期结果与实际结果的对比
3. 发现问题时自行修正

## 组合使用

多个模式可以组合使用以达到更好的效果：

| 信息类型 | 推荐组合 |
|---------|---------|
| 业务流程 | 流程图 + 场景化描述 |
| 多角色协作 | 泳道图 + 序列图 + 判定矩阵 |
| 状态管理 | 状态机 + 极端示例 + 判定矩阵 |
| 业务规则 | 判定矩阵 + 实例化 + 反向验证 |
| 算法逻辑 | 流程图 + 可执行验证 + 极端示例 |

## ASCII 图形规范

在终端环境中，优先使用 ASCII 图形。以下是 10 种标准模板：

### 模板速查

1. **架构图** — 层级结构
   ```
   ┌─ 表现层 ─┐
   ├─ 业务层 ─┤
   └─ 数据层 ─┘
   ```
2. **流程图** — 步骤序列，分支用 `↓ [子步骤]`
   ```
   [A] → [B] → [C]
   ```
3. **布局图** — UI/页面结构
   ```
   ┌──Header──┐
   ├Nav┬Content┤
   └──Footer──┘
   ```
4. **组件通信** — 父子/模块交互
   ```
   [Parent] ──props──→ [Child] ←──emit──
   ```
5. **状态机** — 状态流转
   ```
   [idle] → [loading] → [success]
                ↓
             [error] → [retry] ─┘
   ```
6. **时序图** — 竖线对齐，调用顺序
   ```
   A ──req──→ B ──call──→ C ──query──→ D
   ```
7. **树形图** — 目录/层级
   ```
   src/
   ├── components/
   └── views/
   ```
8. **进度图** — 状态追踪
   ```
   [████░░] 50%  ✅完成 🔄进行中 ⏳待处理 ❌失败
   ```
9. **分支图** — 条件判断
   ```
   {判断} ─┬─ Y → [A]
           └─ N → [B]
   ```
10. **对比图** — 方案比较
    ```
    ┌─方案A─┬─方案B─┐
    │ 优点  │ 优点  │
    │ 缺点  │ 缺点  │
    └───────┴───────┘
    ```

### 符号语义约定

| 符号 | 含义 |
|------|------|
| `[步骤/动作]` | 动作节点 |
| `(状态/节点)` | 状态节点 |
| `{决策/条件}` | 决策节点 |
| `<外部系统>` | 外部依赖 |

### 箭头语义

| 箭头 | 含义 |
|------|------|
| `──→` | 普通流程 |
| `═══→` | 主路径 |
| `···→` | 异步/可选 |
| `──✗` | 失败/中断 |

### 常用绘图符号

```
→ ← ↑ ↓ │ ─ ┌ ┐ └ ┘ ├ ┤ ═ ━ █ ░ ✅ ❌ 🔄 ⏳
```

## 输出格式建议

根据场景选择合适的输出格式：

1. **纯文本输出**：使用清晰的标题层级和列表结构
2. **表格输出**：用于对比、矩阵等形式
3. **代码块输出**：用于展示可执行代码、配置等
4. **ASCII 图形输出**：终端环境首选，参照上方模板速查
5. **Mermaid 图形**：支持渲染时使用
6. **混合输出**：结合以上多种形式

## 重要提醒

1. **形式服务于内容**：选择可视化形式的目的是让信息更容易理解，而非追求形式的复杂
2. **因地制宜**：根据输出场景选择合适的格式（如Markdown、HTML、控制台输出等）
3. **可读性优先**：即使使用图形化表示，也要确保文字说明清晰
4. **降低认知负荷**：将复杂信息拆解为易于理解的单元
